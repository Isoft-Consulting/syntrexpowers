- install Claude config
- Claude SubagentStop hook config is written only when fixture discovery proves the event name and payload/decision contract are accepted
- install Codex config
- Codex install refuses enforcing activation and does not write trusted `~/.codex/hooks.json` when fixture discovery does not prove the installed hook feature flag, hook config path, event names, matcher syntax, command execution, payload, and event-specific decision-output contracts for blocking events
- discovery `--dry-run` emits a non-enforcing hook plan and performs no provider config or install-root mutation
- enforcing plan-only is fixture-gated, writes nothing on readiness failure, and on fixture-proven temporary providers emits selected-output-bound JSON hook plans without provider config or install-root mutation; real enforcing activation refuses to mutate provider configs until the caller passes the explicit blocking-mode acknowledgement flag
- enforcing Codex PermissionRequest hook config is written only when fixture discovery proves the event name, command execution, payload schema, and selected block/deny output contract
- base Codex hook config excludes `PermissionRequest` before fixture proof; conditional snippet merge is tested separately
- malformed selected-output contract records cannot enable conditional hook insertion in discovery mode
- install all
- re-install idempotency
- uninstall removes only strict-mode hooks
- uninstall uses verified manifest hash, validates managed hook entry/output-contract consistency before provider config mutation and after provider filtering, and removes by exact removal selectors rather than substring deletion
- default uninstall updates the install manifest and protected install baseline to the post-uninstall provider config fingerprints and removed managed hook entries before reporting success; stale baselines that still list removed hooks fail uninstall self-check
- default uninstall verifies the protected install baseline, active-runtime-link/target fingerprints, provider config fingerprints, and fixture metadata fingerprints before publishing an uninstall marker; active path tampering or mismatch refuses automatic deactivation
- uninstall writes a pending transaction marker before provider config mutation, uses an empty `staged_runtime_path` plus post-uninstall manifest/baseline candidate hashes, records installer-backup ledger coverage, marks interrupted deactivation as `uninstall-failed`, keeps empty `staged_runtime_path` valid when rollback recovery is already in `rollback-in-progress`, makes hooks fail closed while the marker is unsettled, and recovery resumes idempotently or refuses with repair guidance
- malformed, extra-fielded, permission-unsafe, symlink-confused, or hash-mismatched install manifests make uninstall refuse automatic removal
- managed hook entry records use empty `matcher` only for provider events without matcher fields, require non-empty `output_contract_id` for enforcing block/deny-capable or inject-capable entries, derive that id from `selected_output_contracts` for the same provider/logical event, bind `self_timeout_ms`/`provider_timeout_ms`/`provider_timeout_field` to the generated command or provider-native fields, write hash-bound removal selectors, reject duplicate/drifted selectors and unsorted managed entries, and generate hook env summaries only from the exact managed entries
- existing unrelated hooks preserved
- provider config writers use structured JSON/TOML parsing, reject duplicate keys or malformed configs before merge, preserve unrelated entries, and never use regex/string-splice edits
- installed Claude/Codex hook config files are protected by generated protected-baseline
- generated protected baselines include runtime config plus destructive/protected-path/stub/network/filesystem pattern and allowlist files
- protected install baseline has an exact schema and baseline hash, includes the managed hook entries with timeout fields, is written for staged runtime/config before activation, and is verified again against active disk state after activation
- protected install baseline writes sorted unique file/config record arrays, rejects duplicate or unsorted file/config records, and rejects `provider_config_paths` values that do not exactly match `provider_config_records`
- protected install baseline rejects duplicate fixture manifest tuples, platform mismatches, and non-canonical fixture manifest record ordering
- protected install baseline rejects `protected_file_inode_index` records with invalid kind values or missing path identity fields
- schema and matrix metadata files are copied into the active runtime, included in install manifest `runtime_file_records`, covered by the protected install baseline, parsed under fixed shared-core metadata bootstrap parsers before metadata-driven parsing is trusted, and drift/missing/malformed metadata fails install self-check or enforcing activation
- concurrent install/rollback/uninstall attempts are serialized by exact-schema `state-global.lock/owner.json` records and cannot interleave active symlink, provider config, install-manifest, install baseline writes, or rollback marker phase transitions
- install/rollback/uninstall self-check verifies global trusted-state ledger coverage for install-release, active-runtime-link, install-manifest, runtime-config, protected-config, provider-config, installer-marker, installer-backup, and protected-install-baseline mutations, including symlink-aware active-runtime-link fingerprints and hash-chain continuity
- install transaction marker schema includes previous install manifest/baseline hashes, install versus uninstall staged-field semantics, hash-verified `pre-activation` to `activating` transition, failure promotion to `post-activation-failed`, complete-marker publication, completed-pending cleanup with phase validation before ledger repair, and unresolved-pending refusal before new transactions
- stale `pre-activation` marker recovery verifies unchanged active runtime/config fingerprints before deleting staged files, writes only the limited global repair ledger records allowed by the state-layout writer matrix, is refused by rollback until installer repair promotes it or clears it, interrupted `rollback-in-progress` markers resume idempotently or refuse with repair guidance, rollback resumes after complete-marker publication by validating and reusing the existing complete marker instead of appending duplicate ledger creates, install/rollback/uninstall repair a missing pending-marker delete ledger after delete-before-ledger interruption, cross-writer and duplicate marker ledger records refuse instead of being duplicated, and finalization refuses a missing pending-marker delete preimage instead of accepting a no-op delete
- backup manifest schema records runtime config with exact backup records, requires `existed` on backup file records, validates empty backup paths for missing files and non-empty relative backup paths for existing files, validates exact active-runtime fingerprint objects for missing/symlink/directory cases, rejects invalid backup file kinds, duplicate backup records, and malformed provider/kind coupling, and hash-binds previous install manifest/baseline plus active runtime paths/fingerprints for install rollback and uninstall recovery
- backup manifest rejects existing config records without exactly one matching backup file record, rejects summary arrays that drift from `backup_file_records`, rejects missing config records that still reference backup blobs, rollback verifies every referenced backup blob against its recorded `content_sha256` before writing `rollback-in-progress` or restoring files, and rollback verifies restored file content/mode/size, active symlink target, missing records, and staged runtime deletion before publishing a complete marker
- fresh-install rollback represents missing previous active runtime explicitly and removes only the active runtime symlink created by the failed transaction after verifying the current active symlink target matches `staged_runtime_path`
- legacy active runtime directory activation is rejected by the current discovery skeleton; future migration tests must prove manifest/baseline ownership, exact directory backup, install-release ledger coverage, and rollback restore before enabling it
- active runtime path as a regular file, special file, symlink-confused path, unrecognized directory, or unsafe directory fails activation before provider configs are touched
- rollback restores previous provider configs, install manifest, runtime config, protected config, install baseline, and active runtime package files or symlink when baseline verification fails
- rollback refuses to restore an existing-file backup record onto a symlink or non-file target, so provider config restore cannot follow attacker-created symlinks
- runtime.env whitelist parsing, invalid-value failure behavior, checked-in protected config template validation, and installed worker defaults staying disabled until protected `worker-invocation` proof exists
- install root path containing spaces remains executable through generated config
- generated hook self-timeout delivery works through fixture-proven command env assignment or provider-native hook env field, and any provider-native outer timeout is normalized, protected in the manifest/baseline, and at least 1000 ms greater than the self-timeout
- README provider support matrix claims are parsed through the closed provider-feature status/proof enums and checked against installer feature gates and selected fixture proofs; unknown status/proof text or any row that claims enforcement without matching protected fixture/install evidence fails
- installer feature gates cannot mark a provider capability or global install/runtime capability as enforcing-ready when any touched schema id or matrix id in [Implementation Readiness](../17-implementation-readiness.md) lacks an applicable required artifact: protected metadata, parser or validator entrypoint, valid and malformed fixtures, hash validation when applicable, positive and negative matrix coverage when applicable, or fail-closed tests
- README provider support matrix is present and maps FDR challenge to fixture-gated support only when normalized current-turn assistant text is fixture-proven; unproven providers remain `disabled-until-proof`
