# 8.5 Destructive Gate

Part of [Shared Core Components](../08-shared-core-components.md).


Runs on `pre-tool-use` for normalized `tool.kind=shell`. Before provider block/deny output is fixture-proven, the same classifier may run only as discovery preflight and write a hash-bound `hook.preflight.v1` diagnostic record; that record is log-only and cannot become trusted allow/block evidence.

The gate uses shared pattern files:

- `~/.strict-mode/config/destructive-patterns.txt`
- `~/.strict-mode/config/protected-paths.txt`
- `~/.strict-mode/config/filesystem-read-allowlist.txt`
- `~/.strict-mode/config/network-allowlist.txt`
- project overrides in `<project>/.strict-mode/`

Protected path matching is not optional for the installed strict-mode runtime, provider hook config, state/config paths, legacy opt-out paths when enabled, and project-local `.strict-mode/` overrides. Even when project destructive gate is disabled, writes to protected strict-mode paths remain blocked unless they are performed by installer/uninstaller/admin commands outside provider tool execution.

Protected path command matching:

- For write/edit/multi-edit/patch tools and path-bearing `other` or unknown write-like tools, compare normalized absolute target paths, resolved real paths, and final-file `dev+inode` values against protected roots and the protected file inode index. Existing targets must be checked with `lstat` for every path component and `stat` for the final file; if the final `dev+inode` matches a protected file, block even when the lexical path is outside protected roots. For new files, resolve and `lstat` the nearest existing parent directory, reject protected/symlinked parents, and apply the remaining path lexically. Paths containing `..` are normalized before comparison.
- An enforcing write/edit/multi-edit/patch, path-bearing `other`, or unknown write-like tool whose target path set is missing, partially unknown, or not normalizable blocks before execution with `protected target unknown`. Stop-time dirty fallback is not a substitute for pre-tool protected-root enforcement, because a protected write would already have happened.
- For shell tools, parse shell words without executing expansion, then compare every literal/quoted path token, every cwd-relative path token normalized against the verified hook `cwd`, and safely expanded `~`, `$HOME`, and `${HOME}` forms against all protected-root aliases: the absolute lexical install root, `<install-root>/active`, generated hook command paths, configured state/config roots, provider config paths, project `.strict-mode` paths, and the resolved active runtime target from protected baselines. Redirection targets, `tee` targets, `mv`/`cp`/`rm`/`chmod` operands, and script entrypoint path operands are path tokens for this purpose. Matching must not depend on whether the protected path token was quoted. If a relative token contains `..`, strict-mode normalizes it against cwd and blocks when the result enters a protected root; if cwd identity is untrusted, the command blocks rather than treating relative paths as safe.
- Shell commands that mention a protected root together with write-like operators, mutators, or inline interpreters (`>`, `>>`, `tee`, `mv`, `cp`, `rm`, `chmod`, `chown`, `truncate`, `sed -i`, `perl -pi`, `python -c`, `node -e`, `ruby -e`, `php -r`) block unless the matcher can prove read-only intent.
- If a shell command mentions a protected root and write intent is uncertain, block with a reason asking for manual terminal execution outside the agent.
- Top-level shell commands and commands nested inside command/process substitutions with write capability whose target set cannot be statically proven disjoint from protected roots block. Examples include inline interpreters, generated scripts, `sh -c` strings, package install/update commands, or project scripts with unknown write targets.
- Shell commands that create aliases to protected files or directories, including `ln`, `ln -s`, `cp -al`, bind-mount style commands, or scriptable hardlink/symlink creation, block unless the matcher proves both source and target are outside protected roots and do not match protected `dev+inode` entries.
- Provider shell commands must not execute strict-mode runtime entrypoints or helper scripts directly or through shell command substitution, process substitution, backticks, or wrapper argv. Any command whose executable resolves under the protected install root `bin/`, `core/`, `lib/`, `providers/`, or `templates/` blocks, including `strict-hook`, `strict-judge`, core scripts, provider installers, and scripts reached through symlinks or relative paths.
- The only allowed protected-root write from a provider shell command is the parsed argv tuple `[<install-root>/active/bin/strict-fdr, "import", "--", <source-path>]`, resolving the executable through the protected active symlink, verifying its fingerprint against `protected-baseline`, and accepting `<source-path>` only after the FDR source identity checks in [Stop Guard](04-stop-guard.md) prove it is a regular project file outside protected roots with no symlink component, protected hardlink, device, FIFO, socket, directory, or oversize content. Provider command strings may quote the executable path to produce one argv token, but the trusted comparison is against parsed argv values, not raw quote characters. The command must contain no env assignment prefix, shell wrapper, pipe, redirect, command substitution, glob, process substitution, or extra arguments. If command parsing or source identity cannot prove the exact trusted import subject, the command is blocked and the agent may still produce project markdown for manual import outside the provider tool.
- The `strict-fdr import` exception is also the only allowed provider shell execution of a strict-mode runtime script. Running `strict-judge`, `strict-hook`, `install.sh`, `uninstall.sh`, `rollback.sh`, or any core/lib/provider script from a provider tool is blocked even if the command appears read-only, because these entrypoints can create trusted state or alter enforcement.
- Stop-time protected-root integrity verification is the backstop for symlink or dynamic shell paths that pre-tool lexical matching cannot prove.

Trust boundary:

- This bundle protects against provider-tool writes by the coding agent. It does not claim to protect against out-of-band user edits, malware, or processes with the same OS account that modify the runtime between hook invocations.
- Because a compromised `strict-hook` cannot reliably detect its own compromise, pre-tool protected-path blocking is mandatory and conservative. Integrity verification is a backstop, not the primary defense.

Blocking requires explicit user confirmation through one of:

- confirmation hash file created by `strict-hook user-prompt-submit` after a matching pending destructive block
- active `<project>/.strict-mode/no-destructive-gate` opt-out validated by the baseline and pending-approval rules, and only for project-destructive non-protected patterns covered by the opt-out effect matrix

Destructive commands must not support same-turn agent-created one-shot bypass. The agent can write arbitrary files, so a bypass file alone is not evidence of user consent.

Confirmation hash contract:

- `pre-tool-use` block writes `pending-destructive-<provider>-<sid>-<hash>.json` using the exact pending approval schema for destructive confirmations and appends an exact-schema `destructive-log.jsonl` `blocked` record with `source=pre-tool-hook` in the same mixed global/session transaction.
- Block reason tells the agent to ask the user for the exact phrase `strict-mode confirm <hash>` and then stop.
- On the next `user-prompt-submit`, `strict-hook` inspects the actual user prompt. If a trimmed line exactly equals `strict-mode confirm <hash>` and the hash matches a pending command for the same provider, session key, raw session hash, cwd, project dir, normalized command hash, `command_hash_source="shell-string"`, pending record hash, and exact `next_user_prompt_marker`, the hook creates `confirm-<provider>-<sid>-<hash>`.
- The same hook append writes a matching exact-schema `destructive-log.jsonl` `confirmed` record with `source=user-prompt-hook`.
- Generic affirmations such as `yes`, `ok`, or `да` must not create confirmation files.
- The agent must never be instructed to create the confirmation file itself.
- The confirmation file by itself is never sufficient; allow requires the matching confirmation audit record.
- A confirmation created by `strict-hook user-prompt-submit` from an exact user phrase and matching audit record with `source=user-prompt-hook` may be consumed immediately on the following tool retry. `STRICT_CONFIRM_MIN_AGE_SEC` (default 5 seconds) applies only to pre-existing confirmation files found in the turn baseline; it prevents same-turn file forgery, not legitimate user-prompt-hook approvals.
- The confirmation expires after `STRICT_CONFIRM_MAX_AGE_SEC` (default 600 seconds) and is valid only when the approving `user-prompt-submit` has the pending record's exact `next_user_prompt_marker`. Expired confirmations are ignored for allow. Deleting an expired confirmation marker or pending record is allowed only under the cleanup or first-next-prompt expiry transaction, with matching `expired` audit and trusted-state ledger records; an unaudited delete is protected-state forgery.
- The confirmation file authorizes exactly one allow. Consumption means that before emitting the provider allow decision, `strict-hook` must hold the relevant state locks, verify the matching marker and audit record, atomically rename `confirm-<provider>-<sid>-<hash>` to `consumed-confirm-<provider>-<sid>-<hash>.json`, append a `consumed` record to `destructive-log.jsonl` that binds the marker hash, active marker path, tombstone path, pre-rename marker fingerprint, and post-rename tombstone fingerprint, and append the matching trusted-state ledger mutation. If tombstone rename, audit append, or ledger append fails, the command fails closed rather than returning allow. A consumed confirmation hash must never authorize a second execution.
- The normalized shell command after trimming trailing whitespace is the input to destructive `command_hash` with `command_hash_source="shell-string"`; the user-visible approval hash is `approval_hash` as defined in the Approval hash schema. The retried command must match exactly; paraphrased or expanded commands require a new confirmation.

General non-destructive quality bypasses may still use audited one-shot `bypass-<provider>-<sid>-<hash>` files, but those files follow the global bypass rule: accepted only with matching audit evidence from turn baseline or when created by `strict-hook user-prompt-submit` from a matching pending bypass record. Destructive gate bypass is confirmation-only.

Quality bypass contract:

- A stop-time quality block may write `pending-bypass-<provider>-<sid>-<hash>.json` using the exact pending approval schema for quality bypasses and append an exact-schema `bypass-log.jsonl` `blocked` record with `source=stop-hook` in the same mixed global/session transaction. The content-scope fingerprint digest covers current worktree fingerprints for existing edited paths plus deletion and rename action markers after dirty-snapshot merge.
- Block reason tells the agent to ask the user for the exact phrase `strict-mode bypass <hash>` and then stop.
- On the next `user-prompt-submit`, `strict-hook` inspects the actual user prompt. If a trimmed line exactly equals `strict-mode bypass <hash>` and the hash matches a pending quality block for the same provider, session key, raw session hash, cwd, project dir, block class, block reason hash, edited path list hash, current-turn tool-intent/tool/edit sequence lists, tool-intent/tool/edit log digests, content-scope fingerprint digest, exact `gate_context`, pending record hash, and exact `next_user_prompt_marker`, the hook creates `bypass-<provider>-<sid>-<hash>`.
- The same hook append writes the matching exact-schema `bypass-log.jsonl` `approved` record with `source=user-prompt-hook`.
- Generic affirmations such as `yes`, `ok`, or `да` must not create bypass files.
- The agent must never be instructed to create the bypass file itself.
- A bypass created by `strict-hook user-prompt-submit` from an exact user phrase and matching audit record with `source=user-prompt-hook` may be consumed immediately on the following Stop retry. `STRICT_CONFIRM_MIN_AGE_SEC` applies only to pre-existing bypass files found in the turn baseline.
- The bypass expires after `STRICT_CONFIRM_MAX_AGE_SEC` (default 600 seconds) and is valid only when the approving `user-prompt-submit` has the pending record's exact `next_user_prompt_marker`. Expired bypasses are ignored for allow. Deleting an expired bypass marker or pending record is allowed only under the cleanup or first-next-prompt expiry transaction, with matching `expired` audit and trusted-state ledger records; an unaudited delete is protected-state forgery.
- The bypass file authorizes exactly one allow. Consumption means that before emitting the provider allow decision, `strict-hook` must hold the relevant state locks, verify the matching marker and audit record, atomically rename `bypass-<provider>-<sid>-<hash>` to `consumed-bypass-<provider>-<sid>-<hash>.json`, append a `consumed` record to `bypass-log.jsonl` that binds the marker hash, active marker path, tombstone path, pre-rename marker fingerprint, and post-rename tombstone fingerprint, and append the matching trusted-state ledger mutation. If tombstone rename, audit append, or ledger append fails, Stop remains blocked rather than returning allow. A consumed bypass hash must never authorize a second Stop allow.
- The pending bypass hash binds the bypass to the block class, block reason, edited path list, current-turn tool-intent/tool/edit sequence lists, tool-intent/tool/edit log digests, content-scope fingerprint digest, exact gate-specific `gate_context`, cwd, project dir, provider, session key, and raw session hash. For FDR challenge bypasses, `gate_context` must bind the bypass to the original blocking FDR cycle record hash, scope digest, and artifact hash that produced the block; diagnostic `blocked-reused` records do not change that subject. For FDR artifact validation bypasses, `gate_context` must bind the bypass to artifact state, artifact hash, verdict, and artifact error hash. For stub/static/trivial quality bypasses, `gate_context` must bind to the normalized finding digest and count. Any later tool-intent/tool/edit record, changed file fingerprint, path-scope change, artifact change, finding-list change, new blocking FDR challenge record, or different block reason requires a new pending bypass.
- During Stop, each quality gate first computes the exact would-be block tuple and bypass subject, then checks for a matching approved bypass candidate. Candidate matching alone does not consume the marker. Consumption is deferred until Stop has proven that no non-quality block exists and every quality block that would otherwise block has an exact approved bypass. A final allow-side transaction then removes only those exact quality blocks after successful tombstone consumption and consumed-audit/ledger append. If any destructive, protected-root, install-integrity, provider-mismatch, permission, unknown-payload, missing-baseline, unresolved-intent, state-corruption, or unmatched quality block remains, Stop blocks and no bypass marker is consumed.
- A `bypass-log.jsonl` `blocked` record is also an unresolved blocked Stop scope until a later Stop appends a matching `resolved` record after successfully allowing that exact or superseding scope, or a matching approved bypass is consumed. Approval TTL expiry only expires the pending approval phrase; it does not let `user-prompt-submit` hide the blocked edited scope. If the logs, sequence lists, digests, or content fingerprints needed to recompute an unresolved blocked scope are missing or untrusted, Stop fails closed with `blocked stop scope untrusted`.
- If an unresolved blocked Stop scope still fails a quality gate but its pending approval record is expired, missing, or untrusted, Stop writes a fresh `pending-bypass-*` and matching `bypass-log.jsonl` `blocked` record for the recomputed exact subject; it must not allow merely because the old approval record expired.
- A `resolved` bypass audit record is written only by Stop, under the same mixed global/session transaction as the safe-boundary update, after all quality gates pass without using a bypass for the unresolved scope. It must match an unresolved `blocked` record by provider/session/raw-session/cwd/project, `pending_record_hash`, block class, block reason hash, edited path list hash, original sequence lists, original log digests, original content-scope fingerprint digest, and exact `gate_context`. It must also include `resolved_*` sequence lists, log digests, and content-scope fingerprint for the current superseding scope that passed. The turn-baseline ledger record that advances `last_safe_stop_sequences` must set `related_record_hash` to the allow-side audit batch hash covering every `resolved`, consumed-bypass, and gate-specific `bypassed` cycle record required by that final allow. A missing or mismatched `resolved` record, missing record from the batch hash, or missing/mismatched baseline ledger relation does not resolve the old blocked scope.
