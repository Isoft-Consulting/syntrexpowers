# 5. Provider Verification And Detection

Part of [Strict Mode Universal Experimental - Specification v0](../SPEC.md).


Provider resolution order:

1. Installer-generated argv: `strict-hook --provider claude|codex <event>`
2. Explicit environment override for manual fixture runs: `STRICT_PROVIDER=claude|codex`
3. Hook payload fields:
   - Claude indicators: `session_id`, `transcript_path`, `hook_event_name`, Claude tool names
   - Codex indicators: Codex hook field names verified by fixtures, `CODEX_THREAD_ID`, Codex-specific raw event naming
4. Process environment:
   - Claude-specific env vars if present
   - Codex-specific env vars if present
5. Fallback: `unknown`

Provider argv/env and payload indicators must be cross-checked. If `--provider codex` is passed but payload has strong Claude indicators, `strict-hook` must log `provider mismatch`, use the explicit provider only in log-only Phase 0, and fail closed for every enforcing event until fixtures resolve the mismatch. A provider-mismatched invocation must not create or update trusted state, confirmations, bypasses, opt-out approvals, protected baselines, sequence records, or FDR artifacts outside Phase 0 fixture capture.

Installer-generated configs must always pass `--provider`; auto-detection is a fallback for development, fixture capture, and manually invoked hooks. `STRICT_PROVIDER` is ignored when `--provider` is present. Any provider resolution that does not start from installer-generated argv, including `STRICT_PROVIDER`, payload-only detection, process-environment detection, or `unknown`, is fixture/manual mode: it may capture diagnostics but must not create confirmations, bypasses, opt-out approvals, protected baselines, sequence records, permission decisions, tool/edit records, turn/session baselines, or trusted FDR artifacts. Installed config path is not trusted by itself because the same entrypoint is shared.

Provider detection must write a compact diagnostic line when result is `unknown`. Runtime discovery logs must store only provider-proof summary fields (`provider_arg_source`, `detected_provider`, `provider_detection_decision`, redacted diagnostic text, and `provider_proof_hash`) plus the payload hash. Runtime parser errors must not include raw payload excerpts, prompt text, source text, or secret-like values in JSONL logs or diagnostics.

Opt-in raw payload capture is allowed only when protected runtime config enables it, the provider proof decision is `match`, the payload is not truncated, and the explicit provider is `claude` or `codex`. Provider process environment cannot enable capture or raise the discovery payload cap. Provider mismatch, conflict, unknown detection, malformed JSON, non-object JSON, unsupported provider values, or truncation keep discovery logging hash-only and set `raw_payload_captured=false`.

Phase 0 provider proof artifacts are exact-schema JSON files generated by `tools/verify-provider-payload.rb` or by `tools/import-discovery-fixture.rb` before fixture manifest mutation. A provider proof has exactly these fields and no extras: `schema_version` (`1`), `provider_arg` (`claude`, `codex`, or `unknown`), `provider_arg_source` (`argv`, `env`, `payload`, `manual`, `fixture-import`, or `unknown`), `payload_sha256`, `detected_provider` (`claude`, `codex`, `unknown`, or `conflict`), `decision` (`match`, `mismatch`, `unknown`, or `conflict`), `claude_indicators`, `codex_indicators`, `conflict_indicators`, `fixture_usable`, `enforcement_usable`, `diagnostic`, and `provider_proof_hash`. Indicator arrays are sorted unique string arrays. `provider_proof_hash` is SHA-256 over canonical proof JSON with only `provider_proof_hash` set to an empty string.

`decision="match"` is the only Phase 0 decision that may be used by fixture import. It requires `detected_provider == provider_arg`, `fixture_usable=true`, and `enforcement_usable=false`. `mismatch`, `unknown`, or `conflict` proofs are diagnostics only and must not update fixture manifests. Imported payload-schema fixtures must include the provider proof artifact in `fixture_file_hashes`, and `payload_schema_hash` must bind the provider proof content together with the raw payload shape and normalized-event artifact.
