# 8.4 Stop Guard

Part of [Shared Core Components](../08-shared-core-components.md).


Runs on `stop`.

Checks:

1. current-turn edit scope is computed; existing paths are scan-able and deleted paths remain in FDR scope
2. unresolved current-turn pre-tool intents or missing direct edit records block before empty-scope skip
3. if current-turn edit scope is empty and no unresolved blocked Stop scope exists, skip stub/FDR artifact gates but still run protected-root, pending confirmation, pending bypass, and opt-out checks
4. no stubs remain when edited source files exist
5. FDR artifact existence is enforced when edited changes exist and the artifact gate is enabled
6. artifact validates when present
7. artifact recheck required if open findings exist and any current-turn tool-intent/tool/edit record is absent from the artifact sequence lists or digest coverage
8. quality blocks compute their exact bypass subject before aggregation, but consume matching approved one-shot bypasses only in the final allow-side transaction
9. destructive confirmation files and quality bypass files are not silently reused

Discovery-only transitional flag:

```
STRICT_NO_ARTIFACT_GATE=1
```

Before Phase 5 enforcement, this flag may downgrade missing-artifact and stale-artifact blocks to audited warnings while `strict-fdr import` is not active. It is not valid for enforcing v0.

`STRICT_NO_ARTIFACT_GATE` is an install-time protected discovery setting only. It may be read from generated hook config or `~/.strict-mode/config/runtime.env` when that file is included in `protected-baseline`; provider tool environment, shell exports, project files, and prompt text must not change it for the current turn. Changing the protected setting through provider tools is protected-root tampering. When Phase 5 or enforcing v0 activation is enabled, `STRICT_NO_ARTIFACT_GATE=1` is a protected config error: edited changes without a valid fresh artifact still block Stop unless an approved project quality-gate opt-out explicitly covers the gate. The flag never disables invalid-artifact blocks, protected-root checks, exact approval rules, or trusted artifact import validation.

FDR artifact validation:

- Artifact path is `fdr-<provider>-<sid>.md` under the shared state root.
- FDR artifacts under the shared state root are strict-mode-owned state. Provider tools must not create or edit them directly.
- A provider-written markdown file in the project may be used as source material, but it is not a trusted artifact until the verified active-runtime `strict-fdr import -- <path>` command validates it and imports a normalized artifact into the state root.
- `strict-fdr import` is the planned v0 command for artifact import. In the current discovery skeleton it is unavailable for trusted state creation and returns `trusted-import-unavailable`; the enforcing importer must verify the source markdown, current-turn edit scope, provider/session/raw-session/cwd/project fields, artifact staleness, and protected-root integrity before writing state.
- Before validating or importing an artifact, the enforcing `strict-fdr import` implementation must acquire the per-provider/session state transaction lock and run the same git dirty-snapshot merge used by Stop. Any shell-created edit records discovered by this merge are appended before artifact validation. The import then freezes `coverage_cutoff_tool_intent_seq`, `coverage_cutoff_tool_seq`, and `coverage_cutoff_edit_seq` as the last trusted current-turn tool-intent/tool/edit sequences that the artifact is required to cover, records `imported_at` after the freeze, and writes the normalized artifact under the same transaction.
- The exact provider shell command that performs trusted `strict-fdr import -- <path>` is import provenance, not review subject matter. Its pre-tool intent record may be excluded from artifact coverage only when the command argv, `command_hash`, `command_hash_source`, `source_fingerprint`, `imported_artifact_hash`, provider, session key, raw session hash, cwd, project dir, `import_intent_seq`, and `import_intent_hash` match the artifact's exact-schema `import_provenance` object. The later `post-tool-use` tool record for the import is created after the artifact is written, so Stop may exclude it only when exactly one trusted tool record after the coverage cutoff has `pre_tool_intent_seq` and `pre_tool_intent_hash` equal to the provenance import intent, matching provider/session/raw-session/cwd/project, matching `command_hash` and `command_hash_source="trusted-import-argv"`, and no edit records. Missing, duplicate, mismatched, or edit-producing import tool records make the artifact stale and keep unresolved-intent handling fail-closed. Any other tool-intent, tool, or edit record after the coverage cutoff makes the artifact stale.
- Stop must not append additional covered edit records after an import freeze. During Stop, if a state-root FDR artifact parses far enough to expose provider/session/raw-session/cwd/project, coverage cutoffs, sequence lists, log digests, and exact-schema `import_provenance`, dirty-snapshot refresh runs in import-freeze compare-only mode before full artifact validation. Compare-only mode checks the current filesystem against the frozen import scope and does not append covered edit records. If it discovers any path/status/fingerprint change not already represented at or before `coverage_cutoff_edit_seq`, the artifact is stale and Stop blocks rather than silently extending the reviewed scope. If the artifact cannot be parsed to that minimal freeze metadata, Stop uses the normal dirty-snapshot merge path and the artifact later fails validation.
- `strict-fdr import` accepts exactly one source file argument after `--`: `strict-fdr import -- <source-path>`. No additional args, env assignment prefix, shell wrapper, pipe, redirect, command substitution, glob expansion, or process substitution is allowed for the trusted import path.
- The source path must resolve to a regular file under the current project directory, outside all protected roots, with no symlink component. `strict-fdr` must reject `/dev/*`, `/proc/*`, FIFOs, sockets, devices, directories, hardlinks to protected-root files, paths outside the project, and files over `STRICT_FDR_SOURCE_MAX_BYTES` (default 1 MiB).
- For providers that expose shell commands as strings, the allowlist check for trusted import must use a shell lexer that produces argv tokens without performing expansion. For providers that expose argv arrays, use the argv array directly. Regex-only command matching is forbidden for this decision.
- The trusted import command parsed argv must be exactly `[<install-root>/active/bin/strict-fdr, "import", "--", <source-path>]`, where the first token is the exact unquoted argv token for the absolute lexical active path generated by the installer. Provider command strings may need shell quotes to produce that single token, but quotes are not part of the parsed argv value. After parsing, strict-mode resolves that executable through the protected active symlink and verifies the resolved target and fingerprint against `protected-baseline`; the resolved release path is not accepted as a provider command token and is not substituted into provenance. Paths with spaces are valid only when represented as one argv token. Any control operator, redirection, environment assignment, wrapper command, command substitution, glob metacharacter that would require expansion, or extra token makes the command untrusted.
- Source identity checks must `lstat` every path component to reject symlink components, then `stat` the final file. `protected-baseline` and `protected-install-baseline` store `dev+inode` for protected files; `strict-fdr import` rejects a source whose final `dev+inode` matches any protected file. If the protected inode index is unavailable, files with link count greater than 1 are rejected for trusted import rather than risking a protected-root hardlink bypass.
- The source file is treated as untrusted input. Import must parse and normalize it; it must never execute content from the source file. Source-supplied `provider`, `session_key`, `raw_session_hash`, `cwd`, `project_dir`, `turn_marker`, coverage cutoffs, sequence lists, log digests, path arrays, `imported_at`, and `import_provenance` are ignored and recomputed from trusted hook/import state. The source may contribute reviewer text, `review_generated_at`, verdict, and findings only after schema validation, path-scope validation, and redaction. A source document that already contains a `json strict-fdr-v1` block is still untrusted source material, not a trusted state-root artifact.
- The normalized state-root artifact must start with a fenced JSON block whose info string is exactly `json strict-fdr-v1`. `strict-hook stop` parses only this JSON block; any prose before it makes the artifact invalid, and prose after it is ignored for enforcement.
- The fenced artifact JSON has exactly these top-level fields and no extras: `schema_version` (`1`), `provider`, `session_key`, `raw_session_hash`, `cwd`, `project_dir`, `review_generated_at`, `imported_at`, `turn_marker`, `coverage_cutoff_tool_intent_seq`, `coverage_cutoff_tool_seq`, `coverage_cutoff_edit_seq`, `tool_intent_seq_min`, `tool_intent_seq_max`, `tool_intent_seq_list`, `tool_intent_log_digest`, `tool_seq_min`, `tool_seq_max`, `tool_seq_list`, `tool_log_digest`, `edit_seq_min`, `edit_seq_max`, `edit_seq_list`, `edit_log_digest`, `edited_paths`, `deleted_paths`, `renamed_paths`, `findings`, `finding_count`, `verdict`, `reviewer`, and `import_provenance`. Missing fields, extra fields, duplicate JSON object keys, wrong primitive types, non-canonical path arrays, or values outside the field domains make the artifact invalid.
- `raw_session_hash` is the full lowercase SHA-256 hex digest over canonical JSON `{"provider":"...","raw_session_id":"..."}`, matching the source used to derive the state filename session key.
- `provider`, `session_key`, `raw_session_hash`, `cwd`, and `project_dir` must match the current normalized Stop/import tuple exactly. `turn_marker` is the fixture-proven normalized turn marker or an empty string when turn markers are untrusted; a non-empty artifact turn marker that does not match trusted current-turn filtering makes the artifact stale. `reviewer` is diagnostic only: a UTF-8 string from the normalized source document capped at 128 bytes after redaction, never a freshness or approval input.
- `review_generated_at` is the reviewer/source-document timestamp. It is diagnostic only and must be valid RFC 3339 UTC, but it is not used to prove freshness for dirty-snapshot edit records appended during import. `imported_at` is the strict-mode import timestamp recorded after dirty-snapshot merge and coverage cutoff freeze; it must be valid RFC 3339 UTC and not earlier than `review_generated_at`.
- `import_provenance` is an object with exactly these fields and no extras: `schema_version` (`1`), `provider`, `session_key`, `raw_session_hash`, `cwd`, `project_dir`, `source_path`, `source_realpath`, `source_fingerprint`, `source_size_bytes`, `argv`, `command_hash`, `command_hash_source`, `import_intent_seq`, `import_intent_hash`, `imported_artifact_hash`, `coverage_cutoff_tool_intent_seq`, `coverage_cutoff_tool_seq`, and `coverage_cutoff_edit_seq`. `source_path` is the normalized project-relative source path; `source_realpath` is the absolute resolved path accepted by source identity checks; `source_fingerprint`, `command_hash`, `import_intent_hash`, and `imported_artifact_hash` are full lowercase SHA-256 hex digests. `source_fingerprint` is SHA-256 over canonical JSON containing `source_realpath`, final file `dev`, `inode`, `mode`, `size_bytes`, `mtime_ns`, and lowercase `content_sha256` after source identity checks. `argv` is the exact trusted lexical argv array `[<install-root>/active/bin/strict-fdr, "import", "--", <source-path>]`; the resolved release path is verified against protected baselines but never replaces this array for hashing. The fourth argv token must normalize against `cwd` to exactly `source_path` and resolve to exactly `source_realpath`; otherwise the import provenance is invalid. `command_hash_source` is exactly `trusted-import-argv`. `command_hash` is SHA-256 over canonical JSON for that lexical argv array plus cwd, project dir, provider, session key, raw session hash, and `command_hash_source`. `import_intent_seq` and `import_intent_hash` identify the matching pre-tool intent record for the trusted import command. `imported_artifact_hash` is SHA-256 over the canonical normalized artifact JSON with only `import_provenance.imported_artifact_hash` set to an empty string. The provenance coverage cutoffs must equal the top-level coverage cutoffs. Missing fields, extra fields, non-canonical paths, hash case/length mismatch, or any value mismatch rejects the import-provenance exclusion.
- Sequence fields are non-negative integers. Coverage cutoff fields equal the last trusted current-turn tool-intent/tool/edit sequence numbers frozen by import. `tool_intent_seq_list`, `tool_seq_list`, and `edit_seq_list` are sorted unique integer arrays equal to the current-turn tool-intent/tool/edit sequence sets from trusted logs at or before their corresponding coverage cutoff. The only permitted omissions are the verified trusted-import provenance pre-tool intent record and the single verified post-cutoff trusted-import tool record described by the trusted import provenance rule in this sub-spec. Min/max fields are summaries only; they must equal the first and last value in the corresponding list, or `0` when that list is empty.
- `tool_intent_log_digest`, `tool_log_digest`, and `edit_log_digest` are lowercase SHA-256 hex digests over canonical JSON arrays of the exact normalized current-turn tool-intent/tool/edit log records covered by the artifact, in ascending `seq` order and bounded by the coverage cutoffs. Stop recomputes these digests from trusted logs; mismatch makes the artifact invalid.
- `edited_paths`, `deleted_paths`, and `renamed_paths` are sorted arrays of exact-schema objects with no extras. Edited entries contain `path`, `action` (`"create"` or `"modify"`), and `fingerprint`; deleted entries contain `path`, `action` (`"delete"`), and `old_fingerprint`; renamed entries contain `old_path`, `new_path`, `action` (`"rename"`), `old_fingerprint`, and `new_fingerprint`. Fingerprint objects use the trusted ledger fingerprint schema. These arrays must cover every current-turn edit entry after dirty-snapshot merge. Missing paths, duplicate paths, paths outside reviewed scope, or action/fingerprint mismatches make the artifact invalid.
- `findings` is an array of objects with exactly `severity`, `path`, `line`, `claim`, `evidence`, `impact`, and `recommendation`. `severity` is `critical`, `high`, `medium`, `low`, or `info`; `path` is a normalized project-relative path from the artifact edit scope or an empty string only for cross-file findings; `line` is a positive integer or `0` when no single line applies. If `path` is empty, `line` must be `0`; if `line` is positive, `path` must be non-empty and refer to an edited path that still exists in the reviewed scope. `claim`, `evidence`, `impact`, and `recommendation` are non-empty UTF-8 strings capped at 4096 bytes each after redaction. `finding_count` must equal `findings.length`.
- `verdict` is one of `clean`, `findings`, or `incomplete`. `clean` is valid only when `finding_count` is `0` and `findings` is empty. `findings` requires `finding_count > 0`. `incomplete` always blocks Stop unless an explicit quality bypass is approved.
- Artifact sequence lists and log digests must cover all current-turn tool-intent/tool/edit records at or before the coverage cutoffs, excluding only verified import provenance. `imported_at` and the state-root artifact mtime must not be older than records created by the import transaction or the frozen coverage cutoff as a sanity check. `review_generated_at` is advisory and never proves freshness. Wall-clock time is not sufficient for freshness when sequence coverage or log digests do not match.
- `artifact_error_hash` is a Stop-computed validation value, not a field accepted from the source artifact JSON. It is SHA-256 over a canonical sorted array of stable artifact validation error objects with `code`, `path`, `field`, and redacted `detail_hash`. Missing artifacts use 64 zeroes. Free-form rendered error text is not hashed directly.
- If artifact validation cannot parse the artifact, Stop blocks regardless of `STRICT_NO_ARTIFACT_GATE`.
