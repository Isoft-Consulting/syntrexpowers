# 8.3 Record Edit

Part of [Shared Core Components](../08-shared-core-components.md).


Runs on `post-tool-use`.

Pre-tool intent tracking:

- Before `pre-tool-use` returns allow for any shell, write, edit, multi-edit, patch, path-bearing `other`, or unknown write-like tool, it must append a trusted `tool-intents-<provider>-<sid>.jsonl` record under the session state transaction lock.
- Each tool-intent JSONL record has exactly these fields and no extras: `schema_version` (`1`), `seq`, `provider`, `session_key`, `raw_session_hash`, `cwd`, `project_dir`, `turn_marker`, `logical_event`, `tool_kind`, `tool_name`, `normalized_command_hash`, `command_hash_source`, `normalized_path_list`, `payload_hash`, `write_intent`, `ts`, and `intent_hash`. `normalized_command_hash` is 64 zeroes when no command is present and must equal the matching tool invocation record's `command_hash`. `command_hash_source` is `none`, `shell-string`, `argv`, or `trusted-import-argv`. `normalized_path_list` is a canonical sorted array of normalized absolute paths or an empty array. `write_intent` is `none`, `read`, `write`, or `unknown`; write/edit/multi-edit/patch tools are `write`, read-only tools are `read`, shell/path-bearing `other` tools are `write` only when write intent is proven and `unknown` when capability or target cannot be proven. `payload_hash` is SHA-256 over canonical JSON containing the normalized `tool` and `permission` objects used for the decision, with `raw` omitted and logging-policy redaction/truncation applied; if no payload object exists, it is the hash of `{}`. `intent_hash` is SHA-256 over the canonical intent record with only `intent_hash` set to an empty string.
- If an enforcing `pre-tool-use` event cannot append the intent record for a shell or write-like tool, it fails closed before the provider tool executes. Discovery/log-only phases may log degraded state, but they cannot later treat missing intent records as trusted evidence.
- Each permission decision JSONL record has exactly these fields and no extras: `schema_version` (`1`), `seq`, `provider`, `session_key`, `raw_session_hash`, `cwd`, `project_dir`, `turn_marker`, `logical_event` (`"permission-request"`), `permission_operation`, `requested_tool_kind`, `decision` (`"allow"` or `"deny"`), `reason_code`, `normalized_path_list`, `network_tuple_list`, `payload_hash`, `ts`, `previous_record_hash`, and `record_hash`. `permission_operation` is one of `tool`, `shell`, `write`, `network`, `filesystem`, `combined`, or `unknown`. `requested_tool_kind` is one of `shell`, `write`, `edit`, `multi-edit`, `patch`, `read`, `other`, or `unknown`. `reason_code` is a closed enum: `allow-safe`, `allow-read-only`, `allow-exact-allowlist`, `deny-protected-root`, `deny-destructive`, `deny-network`, `deny-filesystem`, `deny-broad-scope`, `deny-unknown`, `deny-invalid-payload`, `deny-untrusted-state`, `deny-install-integrity`, `deny-fixture-missing`, `deny-record-failure`, or `deny-policy`. `normalized_path_list` is a canonical sorted array whose entries are normalized absolute paths or the exact sentinel string `unknown`; `decision="allow"` is valid only when every path entry is a normalized absolute path. `network_tuple_list` is a canonical sorted array of exact objects with no extra fields: `scheme`, `host`, `port`, and `operation`; `scheme`, `host`, or `operation` may be `unknown` only for `decision="deny"`, and `port` is either an integer `1..65535` or the string `unknown` only for `decision="deny"`. Port `0`, negative ports, non-integer ports, and out-of-range ports are invalid and cannot appear in an allow record. `payload_hash` is SHA-256 over canonical normalized `permission` plus mirrored `tool` data with redaction/truncation. `previous_record_hash` is 64 zeroes for the first record in the provider/session permission-decision log and otherwise equals the preceding trusted permission decision record hash for the same provider/session log. `record_hash` is SHA-256 over canonical permission decision JSON with only `record_hash` set to an empty string.
- Permission decision sequence allocation uses the same session transaction lock and exact-schema monotonic `seq-*` state as tool/tool-intent/edit records, with a distinct sequence domain encoded by record type. Stop and health checks verify permission-decision sequence order and hash-chain continuity for approval-capable providers; missing allow records for approval-capable PermissionRequest events make subsequent native approvals untrusted.
- `post-tool-use` must include the matching pre-tool intent sequence and intent hash in the tool invocation record when the provider exposes enough identity to match it. If exact matching is impossible, Stop treats the current-turn intent as unresolved unless fixture tests prove a safe correlation rule.

For write, edit, multi-edit, patch, path-bearing `other`, and unknown write-like tools, the normalized event must populate `tool.file_changes` when the provider payload, argv, or patch text exposes paths. If only patch text is available, extract every touched path and action from patch headers. Append every resolved change to `edits-<provider>-<sid>.jsonl`. If an enforcing post-tool event for a direct path-bearing tool cannot recover path evidence, cannot append and verify every required edit record, or git dirty-snapshot fallback does not cover the change, the matching pre-tool intent remains unresolved and Stop fails closed.

Edit log format is JSONL, not plain paths. Each edit record has exactly these fields and no extras: `schema_version` (`1`), `seq`, `provider`, `session_key`, `raw_session_hash`, `cwd`, `project_dir`, `turn_marker`, `path`, `old_path`, `new_path`, `action`, `source`, `ts`, and `record_hash`. `turn_marker` is the fixture-proven normalized `turn_id` or an empty string. `record_hash` is SHA-256 over canonical edit record JSON with only `record_hash` set to an empty string.

```json
{"schema_version":1,"seq":42,"provider":"codex","session_key":"...","raw_session_hash":"...","cwd":"/repo","project_dir":"/repo","turn_marker":"...","path":"/abs/file.php","old_path":"","new_path":"","action":"modify","source":"payload","ts":"2026-05-05T12:00:00Z","record_hash":"..."}
```

Deleted paths and rename old paths stay in the log and remain in FDR scope even though stop-time stub scan skips non-existing files. Stop-time checks operate on the current-turn edit scope: entries whose fixture-proven `turn_marker` matches the current normalized event, entries with `seq` greater than the last safe tool-intent/tool/edit sequence boundary recorded in `turn-baseline-<provider>-<sid>.json` when `turn_marker` is absent or unproven, plus any unresolved blocked Stop scope carried by exact-schema `bypass-log.jsonl` `blocked` records for the same provider/session/cwd/project. Timestamps are diagnostic only and must not be the sole current-turn boundary.

Post-tool recording must always append a tool invocation entry before edit entries. Each tool invocation record has exactly these fields and no extras: `schema_version` (`1`), `seq`, `provider`, `session_key`, `raw_session_hash`, `cwd`, `project_dir`, `turn_marker`, `tool_kind`, `tool_name`, `command_hash`, `command_hash_source`, `write_intent`, `payload_hash`, `pre_tool_intent_seq`, `pre_tool_intent_hash`, `ts`, and `record_hash`. `command_hash` is 64 zeroes when no command is present. `command_hash_source` is `none`, `shell-string`, `argv`, or `trusted-import-argv`, and must match the pre-tool intent record. `write_intent` and `payload_hash` use the same domain and canonical input rules as the matching tool-intent record; mismatch makes the tool record untrusted unless fixture tests prove a safe pre/post payload-normalization difference. `record_hash` is SHA-256 over canonical tool record JSON with only `record_hash` set to an empty string.

```json
{"schema_version":1,"seq":41,"provider":"claude","session_key":"...","raw_session_hash":"...","cwd":"/repo","project_dir":"/repo","turn_marker":"...","tool_kind":"shell","tool_name":"Bash","command_hash":"...","command_hash_source":"shell-string","write_intent":"unknown","payload_hash":"...","pre_tool_intent_seq":40,"pre_tool_intent_hash":"...","ts":"2026-05-05T12:00:00Z","record_hash":"..."}
```

The tool log is used to decide whether missing dirty baselines are safe. Post-tool recording must always exit allow, but a current-turn unresolved or untrusted pre-tool intent is a Stop blocker for any shell, write, edit, multi-edit, patch, path-bearing `other`, or unknown write-like tool. This covers direct provider writes in non-git projects where payload-derived paths would otherwise be lost if `post-tool-use` logging failed.

Tool-intent, tool, and edit log current-turn filtering follows the same fixture-proven `turn_marker`, last safe turn-baseline sequence boundary, and unresolved blocked Stop scope carry-over rule. Sequence numbers are monotonic per provider/session and allocated by strict-mode-owned state code before appending each tool-intent, tool, or edit record.

State transaction and sequence allocation contract:

- Allocation uses the exclusive per-provider/session state transaction lock under the state root before reading or writing `seq-<provider>-<sid>.json`, `prompt-seq-*`, `prompt-events-*`, `tool-intents-*`, `permission-decisions-*`, `tools-*`, `edits-*`, or any trusted state file for that provider/session. The lock implementation and stale-lock rules are the shared `state-global.lock/` / `state-<provider>-<sid>.lock/` contract from [State Layout](../07-state-layout.md).
- While holding the lock, state code reads the last committed sequence from exact-schema `seq-*` or `prompt-seq-*` state, increments once per record in the correct record-type domain, writes complete JSONL records to a temporary file or append handle, fsyncs when available, and atomically publishes the updated sequence state. Records and trusted JSON/marker files must never be interleaved, partially appended, or observed half-written.
- Every appended tool-intent, permission-decision, tool, and edit record includes `seq`, provider, session key, raw session hash, cwd, project dir, timestamp, and record hash or intent hash. Duplicate, missing, non-monotonic, hash-mismatched, or unparsable sequence records make the affected log untrusted.
- If sequence allocation, tool-intent append, permission-decision append, tool log append, or edit log append integrity fails, Stop fails closed for any current turn that used shell, write, edit, multi-edit, patch, path-bearing `other`, or unknown write-like tools, permission-request allow decisions fail closed for approval-capable requests, and trusted FDR import is rejected until the log is repaired by a strict-mode-owned repair command.

Shell tools can modify files without exposing paths (`sed -i`, generators, migration tools, formatters, custom scripts). Therefore edit tracking cannot rely only on provider tool payloads.

Dirty snapshot fallback:

1. On `session-start` or the first `user-prompt-submit`, write immutable `session-baseline-<provider>-<sid>.json` if it does not already exist. Enforcing activation requires fixture proof that one of these events runs before any model tool execution for the session; otherwise no trusted dirty baseline, approval evidence, or protected baseline can be created and write-like enforcement for that provider must stay discovery/log-only or fail activation. The baseline captures timestamp, cwd, project dir, provider, session key, raw session hash, current `prompt_seq` when available, fixture-proven turn id when available, initial project opt-out path fingerprints, initial approval evidence snapshots, and current tool-intent/permission-decision/tool/edit log byte offsets and sequence numbers. Existing session baseline files are never refreshed in place.
2. On every `user-prompt-submit`, write `turn-baseline-<provider>-<sid>.json` with timestamp, cwd, project dir, provider, session key, raw session hash, current `prompt_seq`, fixture-proven turn id when available, previous trusted baseline reference, current tool-intent/permission-decision/tool/edit log byte offsets, current last tool-intent/permission-decision/tool/edit sequence numbers, last safe allowed Stop sequence boundaries, unresolved blocked Stop scope references, approval evidence snapshots for confirmation/bypass/opt-out state, and both previous-baseline and current-capture project opt-out fingerprints. The current capture is used for pending opt-out detection and approval evidence, not automatic activation. A `user-prompt-submit` after a blocked Stop must not promote the blocked tool-intent/tool/edit sequences to the safe current-turn cutoff; only a later Stop allow or exact matching approved quality bypass consumption can advance the safe boundary past that blocked scope.
3. On `user-prompt-submit`, capture a git baseline for the current project:
   - tracked dirty files with status and worktree content fingerprint
   - staged files with status and staged blob fingerprint
   - untracked files with worktree content fingerprint
   - ignored files that are visible to the configured ignored-file scan, with worktree content fingerprint
   - deleted files with status marker
   - submodule entries with commit, dirty marker, and nested worktree summary when available
4. On `user-prompt-submit`, capture or refresh the protected-root integrity baseline before the model can run tools.
5. On `post-tool-use`, verify protected-root integrity before accepting any new confirmation, bypass, or opt-out state.
6. On `stop`, capture the same git dirty list again.
7. On `stop`, verify protected-root integrity again before decision aggregation.
8. Add a JSONL edit entry for every path whose status or fingerprint changed between baseline and stop.
9. For non-git projects, direct write/edit/multi-edit/patch payload-derived paths are the only safe source. If a shell tool or unknown write-like tool ran in a non-git project, Stop blocks with `dirty-snapshot unavailable for non-git shell writes`. Protected tree-snapshot support is out of scope for v0 and must not be treated as implicitly available.
10. On successful Stop allow, advance the safe allowed Stop sequence boundaries stored in `turn-baseline-<provider>-<sid>.json` under the session state lock only after all required quality/protected/destructive checks pass and any matching `resolved` or consumed-bypass audit/ledger records have been appended.

Stop-time dirty discovery is authoritative for shell-created edits. Direct PostToolUse logging is an early, best-effort accelerator. Pre-existing dirty files are not pulled into the session scope unless their fingerprint/status changed during the turn.

Ignored files and submodules are fail-closed for shell-created edit scope. If a shell or unknown write-like tool ran and ignored-file enumeration is unavailable, exceeds `STRICT_DIRTY_IGNORED_MAX_FILES` (default 5000), or discovers a changed ignored file not covered by a trusted direct edit record, Stop blocks with `ignored dirty scope untrusted` rather than assuming ignored files are irrelevant. If a submodule status changes and strict-mode cannot capture a nested dirty baseline with the same provider/session/cwd/project tuple rules, Stop blocks with `submodule dirty scope untrusted`. Project quality-gate opt-outs do not disable this scope-safety rule.

Baseline availability:

- `turn-baseline-<provider>-<sid>.json` is required for git dirty-snapshot fallback.
- Current-turn pre-tool intent records are required for every shell or write-like tool allowed by enforcing `pre-tool-use`. If a current-turn intent has no matching trusted `post-tool-use` tool record, Stop blocks with `post-tool record missing for allowed tool`. If a direct path-bearing write intent lacks verified edit records/fallback coverage, Stop blocks with `edit record missing for allowed tool`. Neither case may assume an empty edit scope.
- If `stop` runs in a git project and `tools-<provider>-<sid>.jsonl` shows the current turn used any shell tool or unknown write-like tool but no turn baseline exists, Stop blocks with `dirty-snapshot baseline missing` because shell-created edits cannot be scoped safely.
- If no shell or unknown write-like tool ran, Stop may continue without a turn baseline only when every provider payload-derived edit entry used for current-turn scope has a fixture-proven `turn_marker` matching the current Stop event. If `turn_marker` is absent or unproven and no safe turn-baseline sequence/byte-offset boundary exists, Stop blocks with `turn-boundary missing` rather than guessing the edit scope.
- Non-git projects continue to use payload-derived paths only for direct path-bearing tools. Shell or unknown write-like tools in non-git projects block Stop because their touched paths cannot be scoped safely in v0.
