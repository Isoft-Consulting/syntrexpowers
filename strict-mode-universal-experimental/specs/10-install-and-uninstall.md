# 10. Install And Uninstall

Part of [Strict Mode Universal Experimental - Specification v0](../SPEC.md).


### 10.1 Install Root

Default:

```
~/.strict-mode/
```

The installer copies `bin/`, `lib/`, `core/`, `schemas/`, `matrices/`, `templates/`, and provider fixtures into `<install-root>/releases/<transaction-id>/`. The active runtime root is the resolved `<install-root>/active` symlink target. Existing active runtime files are not overwritten in place during staging. Schema and matrix metadata files are runtime files: they must be included in `runtime_file_records`, protected install baseline fingerprints, active runtime self-checks, and rollback restore verification.

All hook commands written by installer must use the shell-quoted absolute lexical `<install-root>/active/bin/strict-hook` path, preceded only by protected literal environment assignments generated by the installer or represented through fixture-proven provider-native hook environment fields. Discovery commands carry `STRICT_HOOK_TIMEOUT_MS=<milliseconds>` and `STRICT_STATE_ROOT="<state-root>"`. Enforcing blocking commands additionally carry `STRICT_ENFORCING_HOOK=1` and `STRICT_OUTPUT_CONTRACT_ID="<selected-output-contract-id>"` before the hook path, so a running hook can still fail closed when both install manifest and protected install baseline are unreadable. The hook executable path must not be realpath-resolved to `<install-root>/releases/<transaction-id>/bin/strict-hook`, because upgrades and rollback rely on the `active` symlink boundary. Do not rely on `$HOME`, `~`, shell aliases, or provider-specific environment expansion in hook config. Config examples in this sub-spec use `/ABS/HOME` as a placeholder; generated discovery config must contain commands like `STRICT_HOOK_TIMEOUT_MS=60000 STRICT_STATE_ROOT="/ABS/HOME/.strict-mode/state" "/ABS/HOME/.strict-mode/active/bin/strict-hook" --provider codex stop`, while enforcing stop config contains `STRICT_HOOK_TIMEOUT_MS=60000 STRICT_STATE_ROOT="/ABS/HOME/.strict-mode/state" STRICT_ENFORCING_HOOK=1 STRICT_OUTPUT_CONTRACT_ID="codex.stop.block" "/ABS/HOME/.strict-mode/active/bin/strict-hook" --provider codex stop` when command-string env assignment is fixture-proven. The generated self-timeout, generated state root, generated enforcing/output-contract env, and any provider-native outer timeout are managed hook fields in the install manifest and protected install baseline; a mismatch between the command/provider environment, provider timeout field, managed hook entry, and protected baseline makes the hook entry untrusted. Installer tests must include an install root containing a space.

Install transaction ordering:

1. Acquire `state-global.lock/`, verify no other pending install transaction is active for the same install root, preflight the lexical `<install-root>/active` path, and keep the lock until complete-marker publication, complete-marker ledger coverage, pending-marker cleanup, and final self-check succeed or rollback/refusal finishes. The current discovery skeleton classifies the previous active runtime as only missing or symlink before provider config merge or backup-manifest creation. A regular file, special file, symlink-confused path, directory, unrecognized path, or unsafe path at `<install-root>/active` fails here before provider configs are touched. Legacy active runtime directory migration is reserved until directory backup/restore proof exists.
2. Stage package files under a transaction-specific inactive install directory with restrictive permissions.
3. Create or update staged protected `runtime.env` from explicit installer options only; do not import shell environment wholesale.
4. Merge provider configs into staged temporary files while preserving unrelated user entries.
5. Write staged `install-manifest.json` with the exact generated entries and transaction id, then verify its manifest hash.
6. Compute fingerprints and `dev+inode` metadata over the staged runtime, runtime config, protected config, and provider config files, then write a staged protected-install-baseline candidate that references the post-activation resolved active runtime root and the shared `<state-root>/protected-install-baseline.json` target.
7. Run a self-check against the staged runtime, manifest, provider configs, and install baseline. If it fails before activation, discard the staged tree and leave existing hooks/runtime untouched.
8. Write the protected transaction backup and verify its manifest hash.
9. Write the pending install transaction marker in `pre-activation` phase and verify its marker hash.
10. Enter the recoverable activation commit after changing the marker to `activating`: create a new active symlink pointing at `<install-root>/releases/<transaction-id>/`, atomically rename it over `<install-root>/active`, then atomically rename staged provider config and protected config files over their targets in deterministic order. The activation path requires the previous `<install-root>/active` path to be missing or a symlink. A directory at `<install-root>/active` is not migrated by the current discovery skeleton; it is refused before provider configs are touched. Directory migration requires a later proof that can back up, move aside, ledger, and rollback the exact legacy active runtime directory without following symlinks. A regular file, special file, symlink-confused path, unrecognized directory, or unsafe directory at `<install-root>/active` makes activation fail before provider configs are touched. Each rename is atomic, but the multi-file activation is not claimed to be a single filesystem-atomic operation; crash safety comes from the pending marker, backup manifest, hook-side `install transaction in progress` block, and rollback.
11. Publish the staged protected-install-baseline candidate to `<state-root>/protected-install-baseline.json`, append global ledger records for every install-release, active-runtime-link, install-manifest, runtime-config, protected-config, provider-config, installer-marker, installer-backup, and protected-install-baseline mutation in the transaction, then re-read active runtime, runtime config, protected config, and provider config files from disk and verify them against that active install baseline and the global ledger chain. If this post-activation check fails, rewrite the marker to `post-activation-failed` and rollback must restore the previous provider configs, install manifest, runtime config, protected config, install baseline, and active runtime package files or active runtime symlink.

Provider hooks are considered active only after step 11 succeeds, the `.complete.json` marker is published and ledgered, and the pending transaction marker has been removed. If the installer cannot complete the staged baseline write and self-check, it must rollback or leave previous provider hooks untouched; it must not leave a hook config that points at a runtime without a valid active install baseline. Any hook invocation that lands during the activation commit sees the pending marker and fails closed for enforcing events rather than trusting the half-activated runtime/config set.

Install transaction marker contract:

- Before install activation or uninstall deactivation, installer writes `<install-root>/install-transactions/<transaction-id>.pending.json` with directory mode `0700`, file mode `0600`, and no symlink components in the path. Marker writes are temp-file plus atomic rename and parent-directory fsync when available.
- The install transaction marker has exactly these fields and no extras: `schema_version` (`1`), `transaction_id`, `phase` (`"pre-activation"`, `"activating"`, `"post-activation-failed"`, `"rollback-in-progress"`, `"uninstalling"`, `"uninstall-failed"`, or `"complete"`), `install_root`, `state_root`, `staged_runtime_path`, `previous_active_runtime_path`, `previous_install_manifest_hash`, `previous_install_baseline_hash`, `backup_manifest_hash`, `staged_install_manifest_hash`, `staged_install_baseline_hash`, `provider_config_plan_hash`, `created_at`, `updated_at`, and `marker_hash`. `.pending.json` markers may use only unsettled phases: `pre-activation`, `activating`, `post-activation-failed`, `rollback-in-progress`, `uninstalling`, or `uninstall-failed`. `.complete.json` markers must use `phase="complete"`. For install and rollback phases, `staged_runtime_path` is the normalized absolute release path under `<install-root>/releases/<transaction-id>/`, `staged_install_manifest_hash` is the staged post-install manifest candidate hash, and `staged_install_baseline_hash` is the staged post-install baseline candidate hash. For uninstall phases, `staged_runtime_path` must be an empty string, `staged_install_manifest_hash` is the post-uninstall manifest candidate hash, and `staged_install_baseline_hash` is the post-uninstall protected install baseline candidate hash. `previous_active_runtime_path` is the previous active runtime path from the protected backup manifest, not the active runtime target link destination. `previous_install_manifest_hash` and `previous_install_baseline_hash` are the hashes observed immediately before staging activation or uninstall deactivation, or 64 zeroes when the corresponding previous file did not exist. `provider_config_plan_hash` is SHA-256 over the canonical list of provider config paths, pre-transaction fingerprints, staged temp paths, final target paths, and expected post-merge or post-uninstall fingerprints.
- `marker_hash` is SHA-256 over canonical marker JSON with only `marker_hash` set to an empty string. Every phase transition rewrites the marker atomically with a new `updated_at` and marker hash. Unknown phase, path mismatch, hash mismatch, or missing backup manifest hash makes rollback or uninstall recovery refuse automatic recovery.
- Before starting a new install or uninstall transaction under `state-global.lock/`, the entrypoint must scan `<install-root>/install-transactions/*.pending.json`. Before looking up any `.complete.json`, it must verify the pending marker hash, exact schema, `install_root`, `state_root`, and `transaction_id` binding to the pending marker filename. If a pending marker has a matching `.complete.json`, pending-aware cleanup must first verify that the pending phase is one of the completed-operation phases (`activating`, `uninstalling`, or `rollback-in-progress`) before validating the complete marker or appending any repair ledger record. If that valid-phase pending marker has a matching valid `.complete.json`, the entrypoint must validate the complete marker, ensure the complete-marker create exists in the global ledger or append exactly one missing create record for that complete marker, and delete only the pending marker with a global installer-marker delete ledger record attributed to the phase-derived writer of the completed operation (`install`, `uninstall`, or `rollback`). Any existing complete-marker create or pending-marker delete ledger record for the same marker path with a different writer is a mismatch, not a reason to append a second writer's record. After pending-aware cleanup, install, uninstall, and rollback entrypoints must scan `.complete.json` markers whose matching `.pending.json` is absent and repair exactly one missing pending-marker delete ledger record from the prior pending-marker ledger preimage. Complete-only repair must verify that the complete-marker create writer matches the latest non-delete pending-marker ledger writer for that pending path before repairing the delete ledger, so rollback completion cannot be re-attributed to stale install marker records. This complete-only repair must not run while the matching pending marker still exists, because pending-aware validation owns that state. Any remaining pending marker without a valid complete marker blocks the new transaction with rollback/repair guidance before provider configs, manifests, baselines, or active runtime paths are touched.
- After step 11 succeeds, installer publishes `<install-root>/install-transactions/<transaction-id>.complete.json` with `phase="complete"` and a new marker hash, appends or verifies the matching global ledger create mutation, then deletes `<transaction-id>.pending.json` and appends the pending-marker delete mutation before releasing `state-global.lock/`. If the pending marker is already missing at the deletion point, finalization must refuse instead of recording or accepting a missing-to-missing no-op as a delete. No `.pending.json` marker may remain after a successful install or uninstall. Rollback consumes `.pending.json` markers in `activating`, `post-activation-failed`, or `rollback-in-progress` phase. If rollback finds a matching `.complete.json` marker while the `.pending.json` marker still exists, it must validate the complete marker hash, exact schema, `phase="complete"`, and transaction binding fields against the pending marker, then reuse that complete marker and delete only the pending marker; it must not rewrite the complete marker or append a duplicate complete-marker create ledger record. A complete-marker hash, schema, or binding mismatch refuses pending cleanup and leaves the rollback resumable. Uninstall recovery consumes `.pending.json` markers in `uninstalling` or `uninstall-failed` phase and must resume the exact manifest-bound hook removal, manifest rewrite, baseline rewrite, and ledger verification idempotently or refuse with repair guidance. A stale `pre-activation` marker is handled only by installer/repair recovery under `state-global.lock/`: it may delete the staged runtime and pending marker only after verifying the active-runtime-link, provider configs, runtime config, protected config, previous install manifest hash, and previous install baseline hash still match the marker/backup pre-activation fingerprints, and only with the limited repair ledger records allowed by the state-layout writer matrix. If that verification fails, the marker must be promoted to `post-activation-failed` and normal rollback must run rather than silently deleting it.

Installer transaction backup contract:

- Before install activation or uninstall deactivation, installer writes a protected backup under `<install-root>/install-backups/<transaction-id>/` with directory mode `0700` and file mode `0600`.
- The backup manifest has exactly these fields and no extras: `schema_version` (`1`), `transaction_id`, `created_at`, `install_root`, `state_root`, `previous_active_runtime_path`, `previous_active_runtime_kind` (`directory`, `symlink`, or `missing`), `previous_active_runtime_fingerprint`, `previous_install_manifest_hash`, `previous_install_baseline_hash`, `provider_config_records`, `protected_config_records`, `runtime_config_records`, `backup_file_records`, and `manifest_hash`. `directory` is schema-reserved for future legacy active-runtime migration; the current installer refuses active directories before backup-manifest creation.
- `previous_install_manifest_hash` and `previous_install_baseline_hash` are lowercase SHA-256 hashes of the pre-activation or pre-deactivation files, or exactly 64 zeroes when the corresponding file was absent before the transaction. Empty strings are invalid.
- `previous_active_runtime_fingerprint` is an exact object with no extra fields: `exists` (`0` or `1`), `kind` (`"missing"`, `"directory"`, or `"symlink"`), `path`, `link_target`, `dev`, `inode`, `mode`, `size_bytes`, `mtime_ns`, `content_sha256`, and `tree_hash`. When `previous_active_runtime_kind="missing"`, `previous_active_runtime_path` is an empty string, `exists=0`, `kind="missing"`, path/link fields are empty, numeric fields are `0`, and both hashes are 64 zeroes. For `symlink`, `link_target` is the exact normalized symlink target string captured by `readlink`, `content_sha256` is SHA-256 over that link target string, and `tree_hash` is 64 zeroes. For `directory`, `link_target` is empty, `content_sha256` is 64 zeroes, and `tree_hash` is SHA-256 over the sorted exact `active-runtime` backup file record list. The top-level `previous_active_runtime_kind` and `previous_active_runtime_path` must match this fingerprint object.
- Each `provider_config_records`, `protected_config_records`, and `runtime_config_records` entry is exact-schema with no extra fields: `path`, `kind`, `provider`, `existed`, `mode`, `owner_uid`, `dev`, `inode`, `size_bytes`, `content_sha256`, and `backup_relative_path`. Runtime config records use `kind="runtime-config"`, `provider=""`, and `path` equal to the normalized absolute runtime config path. For unscoped protected config records, `kind="protected-config"` and `provider=""`; provider config records use `kind="provider-config"` and `provider` equal to `claude` or `codex`. Each `backup_file_records` entry has the same exact fields: `path`, `kind`, `provider`, `existed`, `mode`, `owner_uid`, `dev`, `inode`, `size_bytes`, `content_sha256`, and `backup_relative_path`. `backup_file_records.kind` is one of `provider-config`, `protected-config`, `runtime-config`, `install-manifest`, `install-baseline`, or `active-runtime`. `existed` is `0` or `1`; unavailable numeric metadata is `0`; missing files use `existed=0`, zero metadata, 64 zeroes for `content_sha256`, and an empty `backup_relative_path`. For existing files, `backup_relative_path` must be normalized relative to the transaction backup directory and must not be empty or contain `..`, symlink components, NUL, newline, or absolute path syntax.
- Every existing provider/protected/runtime config record must have exactly one matching `backup_file_records` entry with the same normalized path, `kind`, `backup_relative_path`, metadata, and content hash. Missing config records must not have backup blobs. Install manifest, install baseline, and active runtime backups are represented only in `backup_file_records`, using `kind="install-manifest"`, `kind="install-baseline"`, or `kind="active-runtime"`.
- `manifest_hash` is SHA-256 over canonical backup manifest JSON with only `manifest_hash` set to an empty string. Rollback and uninstall recovery refuse any backup whose transaction id, manifest hash, file hashes, install root, previous install manifest hash, previous install baseline hash, previous active runtime path, or previous active runtime fingerprint does not match the pending install transaction marker and pre-transaction backup records.
- Rollback restore order is provider configs first to remove broken hook entrypoints, then active runtime package files or active symlink, then runtime config, protected config, install manifest, and install baseline. Before rewriting the marker to `rollback-in-progress`, rollback must verify that the current `<install-root>/active` path is rollback-safe for the transaction: if `previous_active_runtime_kind="missing"`, any current active symlink must point exactly at `staged_runtime_path`; if `previous_active_runtime_kind="symlink"`, any current active symlink must point exactly at either `staged_runtime_path` or the backed-up previous symlink target. Any unrelated active symlink, directory, regular file, or special file refuses recovery before provider configs or marker phase are mutated. If `previous_active_runtime_kind="directory"` appears in a backup manifest, the current rollback skeleton refuses automatic restore because installer-side directory migration is not yet an accepted transaction shape. After restore and staged runtime cleanup, rollback re-reads provider configs, protected config files, runtime config files, install manifest, install baseline, active runtime link, and staged runtime path. It refuses to publish a complete marker if restored files do not match the backup manifest content hash, size, and mode expectations, missing records are still present, the active symlink target does not match the backed-up target, or the staged runtime still exists.

### 10.2 Claude Config Writer

Writes or merges `~/.claude/settings.json`:

Provider config writers must use structured JSON/TOML parsers with duplicate-key rejection for trusted merge decisions. They must preserve unrelated user entries byte-for-byte where the provider format allows stable round-trip; when exact byte preservation is impossible, they must preserve semantic data and record the pre/post fingerprints in the backup manifest. Malformed provider config, duplicate keys in hook-owned paths, unsafe permissions, symlink-confused config paths, or parser round-trip loss of unrelated entries makes install refuse automatic merge with a backup/remediation message. Regex/string-splice config edits are forbidden.

```json
{
  "hooks": {
    "SessionStart": [
      {"hooks": [{"type": "command", "command": "STRICT_HOOK_TIMEOUT_MS=5000 STRICT_STATE_ROOT=\"/ABS/HOME/.strict-mode/state\" \"/ABS/HOME/.strict-mode/active/bin/strict-hook\" --provider claude session-start", "timeout": 6000}]}
    ],
    "UserPromptSubmit": [
      {"hooks": [{"type": "command", "command": "STRICT_HOOK_TIMEOUT_MS=3000 STRICT_STATE_ROOT=\"/ABS/HOME/.strict-mode/state\" \"/ABS/HOME/.strict-mode/active/bin/strict-hook\" --provider claude user-prompt-submit", "timeout": 4000}]}
    ],
    "PreToolUse": [
      {"matcher": ".*",
       "hooks": [{"type": "command", "command": "STRICT_HOOK_TIMEOUT_MS=5000 STRICT_STATE_ROOT=\"/ABS/HOME/.strict-mode/state\" \"/ABS/HOME/.strict-mode/active/bin/strict-hook\" --provider claude pre-tool-use", "timeout": 6000}]}
    ],
    "PostToolUse": [
      {"matcher": ".*",
       "hooks": [{"type": "command", "command": "STRICT_HOOK_TIMEOUT_MS=3000 STRICT_STATE_ROOT=\"/ABS/HOME/.strict-mode/state\" \"/ABS/HOME/.strict-mode/active/bin/strict-hook\" --provider claude post-tool-use", "timeout": 4000}]}
    ],
    "Stop": [
      {"hooks": [{"type": "command", "command": "STRICT_HOOK_TIMEOUT_MS=60000 STRICT_STATE_ROOT=\"/ABS/HOME/.strict-mode/state\" \"/ABS/HOME/.strict-mode/active/bin/strict-hook\" --provider claude stop", "timeout": 61000}]}
    ]
  }
}
```

Claude matcher syntax must be fixture-tested. The generated matcher must cover every fixture-proven shell, write, edit, patch, path-bearing `other`, or unknown write-like tool. A broad matcher such as `.*` is preferred so shared core classification can decide which tools matter. If Claude does not accept a broad matcher in the installed version, the installer must generate the complete fixture-proven tool matcher set and fail activation when required write-like or shell tools cannot be covered.

The base Claude hook config must not include `SubagentStop` until fixture discovery proves Claude accepts that event name and its payload/decision contract for the installed version. If installed, it is merged as a conditional additional event:

```json
{
  "SubagentStop": [
    {"hooks": [{"type": "command", "command": "STRICT_HOOK_TIMEOUT_MS=30000 STRICT_STATE_ROOT=\"/ABS/HOME/.strict-mode/state\" \"/ABS/HOME/.strict-mode/active/bin/strict-hook\" --provider claude subagent-stop", "timeout": 31000}]}
  ]
}
```

### 10.3 Codex Config Writer

The Codex config writer may enter enforcing mode only after provider fixture discovery proves the installed Codex build supports the exact hook feature flag, hook config path, event names, matcher syntax, command execution contract, stdin/stdout payload contract, and decision-output contract selected for install. The selected output contracts for blocking events must be written into the install manifest and protected install baseline as `selected_output_contracts`; `pre-tool-use` and `permission-request` may use block/deny contracts, while `stop` uses a block/continuation contract. Discovery installs write an empty list. If the hook surface is absent, renamed, disabled by policy, or not fixture-proven, the installer may still copy the runtime and record Codex as unsupported/discovery-only, but it must not write `~/.codex/hooks.json` as trusted enforcing configuration and must fail any requested enforcing Codex activation with remediation.

`--enforce --plan-only` is a non-activation planning mode. It must run the same fixture readiness gate and selected-output-contract binding as enforcing activation, emit a JSON plan containing provider config paths, managed hook entries, generated hook commands/env, fixture manifest summaries, and selected output contracts, and must not acquire install locks, copy releases, publish transaction markers, write install manifests/baselines, or mutate provider configs. Real enforcing activation must require an additional explicit blocking-mode acknowledgement flag after plan review, because enforcing `pre-tool-use` and `stop` entries are provider-visible blocking hooks that can interrupt destructive shell commands, deployment workflows, and stop continuation. `--dry-run` is a discovery/non-enforcing alias for plan-only output and must also avoid install/provider writes.

When fixture-proven, it ensures `~/.codex/config.toml` contains:

```toml
[features]
hooks = true
```

When fixture-proven, it merges `~/.codex/hooks.json`:

```json
{
  "hooks": {
    "SessionStart": [
      {"hooks": [{"type": "command", "command": "STRICT_HOOK_TIMEOUT_MS=5000 STRICT_STATE_ROOT=\"/ABS/HOME/.strict-mode/state\" \"/ABS/HOME/.strict-mode/active/bin/strict-hook\" --provider codex session-start"}]}
    ],
    "UserPromptSubmit": [
      {"hooks": [{"type": "command", "command": "STRICT_HOOK_TIMEOUT_MS=3000 STRICT_STATE_ROOT=\"/ABS/HOME/.strict-mode/state\" \"/ABS/HOME/.strict-mode/active/bin/strict-hook\" --provider codex user-prompt-submit"}]}
    ],
    "PreToolUse": [
      {"matcher": "^(Bash|Shell|shell|Write|Edit|MultiEdit|NotebookWrite|NotebookEdit|NotebookMultiEdit|ApplyPatch|apply_patch)$", "hooks": [{"type": "command", "command": "STRICT_HOOK_TIMEOUT_MS=5000 STRICT_STATE_ROOT=\"/ABS/HOME/.strict-mode/state\" \"/ABS/HOME/.strict-mode/active/bin/strict-hook\" --provider codex pre-tool-use"}]}
    ],
    "PostToolUse": [
      {"matcher": "^(Bash|Shell|shell|Write|Edit|MultiEdit|NotebookWrite|NotebookEdit|NotebookMultiEdit|ApplyPatch|apply_patch)$", "hooks": [{"type": "command", "command": "STRICT_HOOK_TIMEOUT_MS=3000 STRICT_STATE_ROOT=\"/ABS/HOME/.strict-mode/state\" \"/ABS/HOME/.strict-mode/active/bin/strict-hook\" --provider codex post-tool-use"}]}
    ],
    "Stop": [
      {"hooks": [{"type": "command", "command": "STRICT_HOOK_TIMEOUT_MS=60000 STRICT_STATE_ROOT=\"/ABS/HOME/.strict-mode/state\" \"/ABS/HOME/.strict-mode/active/bin/strict-hook\" --provider codex stop"}]}
    ]
  }
}
```

Codex matcher syntax must be fixture-tested. Codex `PreToolUse`/`PostToolUse` defaults should target shell + write/edit tool families to avoid high-volume read-only hook noise while still covering enforcement-critical tool calls. If that matcher is not accepted for an installed build, installer must use the broadest accepted matcher that still covers fixture-proven shell/write/edit/patch behavior, or omit matcher where the event permits. Enforcing Codex activation fails when fixture-proven shell, write, edit, patch, path-bearing `other`, unknown write-like tools, or approval-capable `PermissionRequest` events cannot be covered by generated hooks.

The base Codex hook config must not include `PermissionRequest`. An enforcing `PermissionRequest` hook entry is installed only after fixtures prove Codex accepts that event name, executes the command, provides a validated payload schema, and has a selected block/deny output contract. If installed, it is merged as a conditional additional event:

```json
{
  "PermissionRequest": [
    {"hooks": [{"type": "command", "command": "STRICT_HOOK_TIMEOUT_MS=5000 STRICT_STATE_ROOT=\"/ABS/HOME/.strict-mode/state\" \"/ABS/HOME/.strict-mode/active/bin/strict-hook\" --provider codex permission-request"}]}
  ]
}
```

If fixtures prove the event is approval-capable but the deny output contract is not verified, installer may register it only in discovery/log-only mode and must keep that provider out of enforcing v0 activation for risky in-agent approvals. The installer must fail with remediation rather than leaving a configuration where native approvals can bypass strict-mode gates.

If Codex hook config supports timeout fields, the installer must set them as provider-native outer timeouts whose normalized milliseconds are at least `STRICT_HOOK_TIMEOUT_MS + 1000` for the event. If it does not, the generated command must still run through `strict-hook` self-timeout.

### 10.4 Uninstall

Installer writes an install manifest under the resolved install root:

```
~/.strict-mode/install-manifest.json
```

The manifest uses the exact schema and `manifest_hash` contract from [State Layout](07-state-layout.md). Uninstall must reject missing fields, extra fields, malformed hash values, tuple/path mismatch, unsafe permissions, symlink-confused paths, or a `manifest_hash` mismatch before reading removal selectors. Default uninstall must also verify the protected install baseline, current active-runtime-link fingerprint, resolved active runtime target fingerprint, provider config fingerprints, and fixture metadata fingerprints against the manifest/baseline before publishing an uninstall marker. If the active path is missing, regular-file, special-file, symlink-confused, directory where a symlink is expected, or otherwise mismatched to the trusted baseline, default uninstall refuses automatic deactivation and may only offer dry-run recovery guidance.

Uninstall removes only hook entries that exactly match verified manifest records and whose current provider config entry still matches the recorded command, provider-native environment fields, self-timeout, provider-native outer timeout field/value, output contract id, hook event, matcher, and exact removal selector. Before touching provider configs it computes staged post-uninstall provider config files plus post-uninstall manifest/baseline candidates, writes a `.pending.json` transaction marker with `phase="uninstalling"`, and writes a protected backup manifest covering provider configs, install manifest, protected install baseline, runtime config, protected config, and active runtime fingerprints. After removing matched hooks, default uninstall is a deactivation transaction: it writes the exact post-uninstall install manifest and protected install baseline that reflect the post-uninstall provider config fingerprints and no longer list removed managed hook entries, appends global ledger records for the provider-config, install-manifest, protected-install-baseline, installer-marker, and installer-backup mutations, then verifies the updated baseline before publishing the `.complete.json` marker and deleting the pending marker. It does not delete the active runtime release, schema/matrix metadata, runtime config, protected config, logs, or user templates by default. If uninstall fails after publishing the pending marker but before complete-marker publication begins, it rewrites the marker to `uninstall-failed`; if provider configs were already mutated, uninstall recovery must either complete the exact post-uninstall manifest/baseline/ledger commit or refuse with repair guidance. Subsequent hooks fail closed on the pending marker until uninstall recovery completes or refuses. If the manifest is missing or untrusted, uninstall may offer a dry-run recovery mode that lists candidate strict-mode hook commands, but it must not remove substring matches automatically.

When uninstall refuses because the manifest, protected baseline, active runtime, or provider config state is untrusted, operators must preserve the evidence needed by recovery. Do not bypass the refusal by deleting `<install-root>` (for example `rm -rf ~/.strict-mode`) and restoring only one provider config from a backup. That creates split-brain provider state: a still-registered provider hook, such as an orphaned `~/.codex/hooks.json` command, can continue to invoke the deleted `<install-root>/active/bin/strict-hook` path and fail with exit 127. Recovery must instead use the manifest/baseline-bound uninstall, rollback, or repair path; if manual provider cleanup is unavoidable, every strict-mode hook entry for every installed provider must be removed by exact command/selector before the runtime directory is deleted or provider sessions are restarted.

Full wipe is a separate explicit command and must have its own manifest/baseline/ledger contract before it can delete active runtime releases, install backups, logs, templates, or trusted state.

`rollback.sh` consumes only the protected backup whose transaction id, manifest hash, previous install manifest hash, and previous install baseline hash match the pending install transaction marker. Before mutating files it rewrites the marker to `phase="rollback-in-progress"` under `state-global.lock/` and appends the matching installer-marker ledger record. It restores provider config files, protected config files, install manifest, runtime config, install baseline, and active runtime package files or active runtime symlink to the previous verified state. If rollback is interrupted in `rollback-in-progress`, a later `rollback.sh` must resume from the same marker and backup manifest, re-read every target fingerprint, and complete idempotently or refuse with repair guidance; if interruption happened after complete-marker publication but before pending-marker cleanup, resume validates and reuses the existing complete marker instead of rewriting history. If interruption happened after pending-marker deletion but before the pending-marker delete ledger append, the next install/rollback/uninstall entrypoint validates the complete marker and repairs the missing delete ledger from the prior pending-marker ledger preimage instead of appending duplicate complete-marker creates. Hooks continue to block while the marker is unsettled. Rollback must not delete arbitrary files under user projects or state logs, and it must refuse to run when the backup manifest does not exactly match the pending install transaction.
