# 7. State Layout

Part of [Strict Mode Universal Experimental - Specification v0](../SPEC.md).

Rules:

- Include provider in filenames to avoid cross-runtime collisions.
- The `<session_id>` placeholder in state filenames is a collision-safe session key, not the raw provider session id. It is built as `<safe-prefix>-<sha256>`, where `safe-prefix` is the raw session id sanitized to `[A-Za-z0-9_-]` and truncated to 48 characters, and `sha256` is the full lowercase SHA-256 over canonical JSON `{"provider":"...","raw_session_id":"..."}`. If a legacy or truncated key is encountered, strict-mode must verify a stored full `raw_session_hash`; a mismatch makes the state file untrusted. If the raw id is empty, trusted state is unavailable under the stable-session rule in the next bullet.
- Trusted state requires a stable provider session id. Fixture tests must prove that `session_id` or a provider-specific equivalent is stable across `user-prompt-submit`, tool hooks, and `stop` for the same session. If no stable session id is available, fallback hashes are allowed only for diagnostic logs in discovery mode; confirmations, bypasses, opt-out approvals, dirty baselines, protected baselines, and FDR artifacts are disabled or fail closed. Destructive/protected operations block with remediation to run manually outside the agent.
- The installed strict-mode runtime and provider hook configuration are protected roots. By default this includes the entire resolved install root `~/.strict-mode`, `~/.claude/settings.json`, `~/.codex/hooks.json`, `~/.codex/config.toml`, `<project>/.strict-mode`, enabled legacy opt-out paths, and any configured install/state/config/provider-config root. This covers `bin/`, `lib/`, `core/`, `providers/`, `templates/`, `state/`, `config/`, and `install-manifest.json`. Provider tools must not write, delete, chmod, truncate, or shell-redirect into these paths.
- Runtime/config/state protection is enforced in `pre-tool-use` for write/edit/multi-edit/patch tools, path-bearing `other` or unknown write-like tools, permission requests, and shell commands before any bypass, confirmation, or opt-out state is trusted. If protection status is unknown, confirmations, bypasses, and opt-out approvals fail closed.
- Protected root integrity is checked after provider tool execution and at Stop. Installer writes `protected-install-baseline.json` as the trusted post-install bootstrap record. `session-start` or the first `user-prompt-submit` must verify that install baseline before writing `protected-baseline-<provider>-<sid>.json` with fingerprints for immutable runtime files, provider hook config files, project strict-mode config files, protected runtime config, protected pattern/allowlist files, and the allowed trusted-state path classes. The protected baseline must not hash itself and must not store moving content fingerprints for active mutable JSONL logs, pending files, confirmation markers, FDR artifacts, installer transaction markers/backups, provider-owned hook trust state, or other trusted state whose contents legitimately change during a session. Mutable trusted-state integrity is represented by `trusted-state-ledger-<provider>-<sid>.jsonl` and `trusted-state-ledger-global.jsonl`.
  Each ledger JSONL record has exactly these fields and no extras: `schema_version` (`1`), `ledger_scope` (`"session"` or `"global"`), `writer` (`"strict-hook"`, `"strict-judge"`, `"strict-fdr"`, `"strict-worker"`, `"install"`, `"rollback"`, `"uninstall"`, `"cleanup"`, or `"repair"`), `provider`, `session_key`, `raw_session_hash`, `cwd`, `project_dir`, `target_path`, `target_class`, `operation` (`"create"`, `"modify"`, `"append"`, `"rename"`, `"delete"`, or `"checkpoint"`), `old_fingerprint`, `new_fingerprint`, `related_record_hash`, `ts`, `previous_record_hash`, and `record_hash`. Global ledger records use empty strings for provider/session/cwd/project fields unless the global mutation is tied to a specific session approval; session ledger records must populate the full provider/session/raw-session/cwd/project tuple. `target_class` is one of `sequence`, `prompt-sequence`, `prompt-event-log`, `tool-intent-log`, `permission-decision-log`, `tool-log`, `edit-log`, `baseline`, `ledger`, `fdr-artifact`, `fdr-cycle-log`, `nested-token`, `worker-context-pack`, `worker-invocation-log`, `worker-result-log`, `pending-approval`, `approval-marker`, `consumed-tombstone`, `approval-audit-log`, `installer-marker`, `installer-backup`, `install-manifest`, `install-release`, `active-runtime-link`, `provider-config`, `runtime-config`, `protected-config`, or `protected-install-baseline`. `active-runtime-link` is used only for lexical `<install-root>/active` symlink creation, replacement, or removal; release tree file changes remain `install-release`.
  `old_fingerprint` and `new_fingerprint` use the trusted ledger fingerprint schema: exact canonical JSON objects with no extras containing `exists` (`0` or `1`), `kind` (`"missing"`, `"file"`, `"directory"`, or `"symlink"`), `dev`, `inode`, `mode`, `size_bytes`, `mtime_ns`, `content_sha256`, `link_target`, and `tree_hash`. Missing sides use `exists=0`, `kind="missing"`, numeric metadata `0`, empty `link_target`, and 64 zeroes for both hashes. Regular files use `kind="file"`, empty `link_target`, 64 zeroes for `tree_hash`, and `content_sha256` over file bytes. Symlinks use `kind="symlink"`, `link_target` equal to the exact normalized `readlink` target string, `content_sha256` over that link target string, and 64 zeroes for `tree_hash`; symlink fingerprints are captured with `lstat` and must not silently follow the link. Directories use `kind="directory"`, empty `link_target`, 64 zeroes for `content_sha256`, and `tree_hash` only when the field's contract names an exact sorted child-record list; otherwise `tree_hash` is 64 zeroes and cannot substitute for required per-file ledger records. For `active-runtime-link`, `target_path` must be the lexical `<install-root>/active` path and fingerprints may only be `missing` or `symlink`. `related_record_hash` is the audit/artifact/cycle/marker hash that caused the mutation, an allow-side audit batch hash where a field contract says multiple audit records jointly cause one baseline mutation, or 64 zeroes when none applies. For `operation="checkpoint"` ledger records, `related_record_hash` must be 64 zeroes; the non-circular binding runs from the `state.checkpoint.v1` record's `source_ledger_record_hash` back to this ledger record. `record_hash` is SHA-256 over canonical ledger record JSON with only `record_hash` set to an empty string.
  Ledger scope/writer/target/operation combinations are closed, not independent enums. Session-scoped target classes are `sequence`, `prompt-sequence`, `prompt-event-log`, `tool-intent-log`, `permission-decision-log`, `tool-log`, `edit-log`, `baseline`, `fdr-artifact`, `fdr-cycle-log`, `nested-token`, `worker-context-pack`, `worker-invocation-log`, `worker-result-log`, `pending-approval`, `approval-marker`, and `consumed-tombstone`; they require `ledger_scope="session"` and the full provider/session/raw-session/cwd/project tuple. Global install target classes are `installer-marker`, `installer-backup`, `install-manifest`, `install-release`, `active-runtime-link`, `provider-config`, `runtime-config`, `protected-config`, and `protected-install-baseline`; they require `ledger_scope="global"` and an empty tuple unless a field contract explicitly ties the global record to a session approval. `approval-audit-log` is global-scoped and may carry the session tuple only for the exact approval audit record named by `related_record_hash`. Checkpointable target classes are only `prompt-event-log`, `permission-decision-log`, `ledger`, and `fdr-cycle-log`; `ledger` is checkpoint-only and may be session or global according to the checkpointed ledger file's scope. `strict-hook` may write session sequence/log/baseline, FDR cycle, pending approval, approval marker, consumed tombstone, and global approval-audit-log records. `strict-fdr` may write only the session sequence/edit-log records needed for dirty-snapshot import plus the imported `fdr-artifact`. `strict-judge` may write only `nested-token` records. `strict-worker` may append only session worker context-pack, invocation, and result records, with advisory-only related-record binding. `install`, `rollback`, and `uninstall` may write only global install target classes. `cleanup` may append only `expired` approval-audit-log records with `source="cleanup"` plus the matching global ledger append record for expired pending approval or approval marker state, may delete those expired pending approvals or approval markers with session-scoped ledger records, and may delete expired nested tokens with session-scoped ledger records based on token TTL without approval-audit-log records; it must not create or modify approval evidence. `repair` may write only matching checkpoint records and ledger records with `operation="checkpoint"` for checkpointable target classes, plus verified stale pre-activation `installer-marker`, `installer-backup`, and uncommitted staged `install-release` deletes with `related_record_hash` set to 64 zeroes; it must not create approval markers, pending approvals, FDR artifacts, worker records, install releases, active-runtime-link records, provider configs, or protected baselines. Append-only logs use `operation="append"` except strict-mode-owned checkpoint compaction, which must use `operation="checkpoint"` with a matching exact-schema `state.checkpoint.v1` record; consumed tombstones use `operation="rename"`, active runtime link records use `create`, `modify`, `rename`, or `delete` only on the lexical active symlink, and any combination outside this matrix is malformed and cannot attribute a protected-state change.
  The ledger append is verified by hash-chain continuity and is not itself required to create a second self-target mutation record. Stop verifies ledger chain continuity, file permissions, writer allowlist, target path class, exact record schema, and that every mutable protected-state change since the previous integrity check is represented by a valid ledger record. Any protected-root change not attributed to a strict-mode-owned ledger record is a state-forgery finding: subsequent confirmations, bypasses, and opt-outs are ignored and Stop blocks with remediation. The only non-forgery exception is an out-of-band project opt-out file create/modify/delete under `<project>/.strict-mode/disabled`, `<project>/.strict-mode/no-static-prepass`, `<project>/.strict-mode/no-destructive-gate`, or enabled legacy opt-out path when the change was not caused by a provider tool. Creates and modifications are not trusted immediately and follow the pending opt-out approval flow; deletes deactivate only that opt-out and cannot create approval evidence. Provider-tool attempts to create, modify, delete, chmod, or symlink opt-out files are always protected-root violations before this exception is considered.
- The install manifest and install baseline are exact-schema JSON files with no extra fields. `install-manifest.json` has exactly these top-level fields: `schema_version` (`1`), `transaction_id`, `install_root`, `active_runtime_link`, `active_runtime_target`, `state_root`, `config_root`, `package_version`, `managed_hook_entries`, `runtime_file_records`, `runtime_config_records`, `provider_config_records`, `protected_config_records`, `fixture_manifest_records`, `selected_output_contracts`, `created_at`, `updated_at`, and `manifest_hash`. `manifest_hash` is SHA-256 over canonical manifest JSON with only `manifest_hash` set to an empty string. Nested install manifest and install baseline record schemas are defined in the install manifest/baseline nested-array bullet in this file and are part of the same exact-schema contract.
- The install baseline is not trust-on-first-use from the first agent turn. It is written only by `install.sh` after package copy, runtime config creation, manifest write, provider config merge, active-runtime-link activation, and post-activation verification complete successfully. The record lives at `<state-root>/protected-install-baseline.json` and has exactly these top-level fields with no extras: `schema_version` (`1`), `kind` (`"protected-install-baseline"`), `transaction_id`, `install_root`, `active_runtime_link`, `active_runtime_target`, `state_root`, `config_root`, `provider_config_paths`, `managed_hook_entries`, `generated_hook_commands`, `generated_hook_env`, `package_version`, `install_manifest_hash`, `runtime_file_records`, `runtime_config_records`, `provider_config_records`, `protected_config_records`, `fixture_manifest_records`, `selected_output_contracts`, `protected_file_inode_index`, `created_at`, `updated_at`, and `baseline_hash`. `baseline_hash` is SHA-256 over canonical baseline JSON with only `baseline_hash` set to an empty string. The active runtime target must be the resolved target of the lexical `<install-root>/active` symlink, while generated hook commands must still point at the lexical active symlink path. `protected_file_inode_index` includes protected runtime files, schema/matrix metadata runtime files, config, provider-config, pattern/allowlist, install manifest, and fixture manifest files except provider-owned mutable trust-state carrier files such as Codex `~/.codex/config.toml` after removing `[hooks.state]` tables from its stable content hash; loaders still add the current verified inode for those files to the in-memory protected inode set so hardlink aliases remain blocked. If the first hook invocation finds the install manifest or install baseline missing, malformed, extra-fielded, hash-mismatched, path-mismatched, mode-unsafe, symlink-confused, fixture-drifted, or inconsistent with current disk fingerprints, protected status is unknown and protected/destructive operations, confirmations, bypasses, opt-out approvals, and trusted FDR import fail closed with a repair/reinstall message.
- Install manifest and protected install baseline nested arrays are exact schemas with no extra fields. `managed_hook_entries` entries contain `provider`, `provider_version`, `config_path`, `hook_event`, `logical_event`, `matcher`, `command`, `provider_env`, `self_timeout_ms`, `provider_timeout_ms`, `provider_timeout_field`, `output_contract_id`, `enforcing`, and `removal_selector`; `provider` is `claude` or `codex`, `hook_event` is the provider-native hook event name, `logical_event` is the strict-mode logical event, `matcher` is the exact provider-native matcher string or an empty string for events without a matcher field, and `output_contract_id` is the selected fixture `decision-output` contract id or an empty string only for discovery/log-only entries that neither block/deny, stop-continue, nor inject output. Enforcing block/deny-capable entries, stop-continuation-capable entries, and inject-capable prompt entries require a non-empty `output_contract_id`, and the generated command for that entry must carry matching `STRICT_ENFORCING_HOOK=1` and `STRICT_OUTPUT_CONTRACT_ID="<output_contract_id>"` assignments. `provider_env` is an exact object map of generated protected hook environment key/value pairs. `self_timeout_ms` must equal the protected hook timeout policy for that logical event and must match the `STRICT_HOOK_TIMEOUT_MS` command assignment or fixture-proven provider-native hook environment field. `provider_timeout_ms` is the normalized provider-native outer timeout in milliseconds, or `0` when no provider timeout field is installed; nonzero values must be at least `self_timeout_ms + 1000` and require a non-empty `provider_timeout_field`. `provider_timeout_field` is the normalized provider config field selector that carries the provider-native outer timeout, or an empty string only when `provider_timeout_ms=0`. `removal_selector` is an exact object with no extra fields: `provider`, `config_path`, `hook_event`, `matcher`, `command`, `provider_env_hash`, `self_timeout_ms`, `provider_timeout_ms`, `provider_timeout_field`, `output_contract_id`, and `entry_hash`; `provider_env_hash` is SHA-256 over canonical `provider_env`, `entry_hash` is SHA-256 over canonical managed hook entry JSON with only `removal_selector.entry_hash` set to an empty string, and uninstall may remove only an entry whose current provider config re-normalizes to the same selector. `managed_hook_entries` are sorted by `provider`, `config_path`, `hook_event`, `matcher`, `command`, `provider_env_hash`, `self_timeout_ms`, `provider_timeout_ms`, `provider_timeout_field`, and `output_contract_id`; duplicate removal selector tuples are rejected. File/config records contain `path`, `realpath`, `kind` (`"runtime-file"`, `"runtime-config"`, `"provider-config"`, `"protected-config"`, `"fixture-manifest"`, or `"install-manifest"`), `provider`, `exists`, `mode`, `owner_uid`, `dev`, `inode`, `size_bytes`, and `content_sha256`; unavailable numeric metadata is `0`, not omitted. For Codex `~/.codex/config.toml`, `content_sha256` is computed from the file after stripping `[hooks.state]` and `[hooks.state.*]` tables, because Codex owns that trust cache; all other sections, including `[features] hooks`, remain hash-protected. `runtime_file_records`, `runtime_config_records`, `provider_config_records`, and `protected_config_records` are sorted by `path` plus `kind` and reject duplicate path/kind pairs. In the protected install baseline, `provider_config_paths` is a sorted unique array of normalized absolute paths exactly equal to the `path` values from `provider_config_records`; `install-manifest.json` has no `provider_config_paths` top-level field. `fixture_manifest_records` entries contain `provider`, `provider_version`, `provider_build_hash`, `platform`, `event`, `contract_kind`, `contract_id`, `fixture_record_hash`, and `fixture_manifest_hash`; they are sorted by `provider`, `platform`, `event`, `contract_kind`, and `contract_id`, and reject duplicate tuples. `platform` must equal the platform string from the selected fixture manifest record and the current install self-check platform. `generated_hook_commands` is a sorted array of exact `provider`, `hook_event`, `logical_event`, and `command` objects derived from `managed_hook_entries`; `generated_hook_env` is an object map of protected generated hook environment key/value summaries, including timeout, state root, enforcing marker, and output contract id command-prefix sources. `protected_file_inode_index` is an object map keyed by `dev:inode`; each entry contains `content_sha256`, `dev`, `inode`, `kind`, `path`, and `provider`. `kind` is one of `runtime-file`, `runtime-config`, `provider-config`, `protected-config`, `fixture-manifest`, or `install-manifest`; provider-owned mutable trust-state carrier files are excluded from this inode index. Any nested missing field, extra field, invalid enum, duplicate selector, non-canonical path, malformed hash, generated hook env mismatch, generated hook timeout mismatch, generated enforcing/output-contract command mismatch, unexpected baseline-only fields in the install manifest, unexpected manifest-only fields in the protected install baseline, or mismatch between manifest and baseline nested records makes install integrity untrusted.
- Install, rollback, and uninstall mutations of install releases, active-runtime-link, install manifest, runtime config, protected config, provider configs, installer transaction markers, backup manifests, and protected install baseline must append matching `trusted-state-ledger-global.jsonl` records under `state-global.lock/` before reporting success. Non-final file/config/runtime mutations must be ledgered before the `.complete.json` marker is published. Complete-marker finalization publishes and ledgers the `.complete.json` marker, deletes the matching `.pending.json` marker, and ledgers that pending-marker delete before success; if a crash lands after complete-marker publication or after pending-marker deletion, the next install/rollback/uninstall entrypoint may repair only the missing complete-marker create or pending-marker delete ledger record after validating marker hashes, filename/root bindings, transaction writer attribution, and prior marker-ledger preimages. For global install state, cleanup and repair may touch only the global target classes allowed by the closed ledger matrix above; they are not alternate writers for active runtime, provider config, runtime config, protected config, install manifest, active-runtime-link, or protected install baseline changes. This does not grant or remove cleanup's separate session-scoped expiry writes for pending approval, approval marker, or nested-token state, which remain governed by the closed ledger matrix and session lock. The final self-check for install/rollback/uninstall verifies global ledger hash-chain continuity and that every published mutation in the transaction id has a matching ledger record with the correct target class, old/new fingerprint, and related marker/manifest hash. Missing ledger coverage makes the transaction incomplete and forces rollback or repair guidance.
- If `strict-hook` sees any `.pending.json` install transaction marker in `pre-activation`, `activating`, `post-activation-failed`, `rollback-in-progress`, `uninstalling`, or `uninstall-failed` phase, it treats install integrity as unsettled. Enforcing `pre-tool-use`, `permission-request`, and `stop` events block with `install transaction in progress`; confirmation, bypass, opt-out approval, and trusted FDR import state is not accepted. Non-enforcing health/prompt events may log the condition and return the safest provider-compatible allow or warning. This closes the activation and deactivation windows between active-runtime-link/config renames and baseline publication.
- Missing protected baseline after any provider tool execution is fail-closed for Stop and for any confirmation, bypass, or opt-out approval. The hook must block with `protected-baseline missing` rather than trusting state created during the turn.
- Strict-mode-owned state writes happen only inside `strict-hook`, `strict-judge` nested token creation/cleanup, intent-gated `strict-fdr import` artifact import, core scripts invoked by those entrypoints, and installer/admin commands (`install.sh`, `rollback.sh`, `uninstall.sh`, `cleanup`, or `repair`) running outside provider tool execution under the required global/session locks for their target classes. Agent-created attempts to modify runtime/config/state are blocked and logged as state-forgery attempts. Provider tools may create trusted FDR state only through exact `strict-fdr import -- <path>` after the enforcing hook records a matching trusted pre-tool intent; direct provider writes to state remain protected-state forgery.
- All session-scoped trusted state writes must run inside one exclusive `state-<provider>-<sid>.lock/` transaction. This includes `seq-*`, `prompt-seq-*`, `prompt-events-*`, `checkpoints-<provider>-<sid>.jsonl`, `tools-*`, `tool-intents-*`, `permission-decisions-*`, `edits-*`, `session-baseline-*`, `turn-baseline-*`, `dirty-baseline-*`, `trusted-state-ledger-*`, `protected-baseline-*`, FDR artifacts, `fdr-cycles-*`, nested judge tokens, pending approval records, confirmation markers, bypass markers, and consumed confirmation/bypass tombstones. Writes to unscoped global trusted state, including `protected-install-baseline.json`, `trusted-state-ledger-global.jsonl`, `checkpoints-global.jsonl`, `optout-log.jsonl`, `bypass-log.jsonl`, `destructive-log.jsonl`, install manifests, install-release activation, runtime config, protected config, provider config merges, installer transaction markers, and installer backup manifests, must acquire `state-global.lock/`. `install.sh`, `rollback.sh`, `uninstall.sh`, and repair commands that touch global install state hold `state-global.lock/` for the whole transaction from preflight through final self-check or rollback/refusal; they must not publish provider hooks, active-runtime-link changes, manifest changes, install baseline changes, protected config changes, or provider config rewrites outside that lock. If one transaction touches both global and session-scoped state, it must acquire `state-global.lock/` first and then `state-<provider>-<sid>.lock/`; no other lock ordering is allowed. Transactional JSON/marker writes use temporary files in the same directory, fsync the file when available, atomic rename into place, and fsync the parent directory when available. JSONL audit, checkpoint, and ledger writes append complete newline-terminated records under the relevant lock and reject partial, interleaved, duplicate, missing, or unparsable records as untrusted state. Sequence allocation is part of the session-scoped transaction, not a separate lock domain.
- `state-global.lock/` and `state-<provider>-<sid>.lock/` use the same lock primitive contract: atomic directory creation or `O_EXCL` file creation on the target filesystem. Advisory-only locking is not sufficient for trusted state. Each lock contains an exact-schema owner JSON record with no extra fields: `schema_version` (`1`), `lock_scope` (`"global"` or `"session"`), `provider`, `session_key`, `raw_session_hash`, `cwd`, `project_dir`, `transaction_kind`, `pid`, `process_start`, `created_at`, `timeout_at`, and `owner_hash`. Global locks use empty strings for provider/session/cwd/project fields unless the transaction is tied to a specific session; session locks must populate the full tuple. `pid` is a non-negative integer or `0` when unavailable, `process_start` is an OS-specific stable process-start token or an empty string when unavailable, and `transaction_kind` is one of `install`, `rollback`, `uninstall`, `cleanup`, `repair`, `prompt-event`, `pre-tool-intent`, `permission-decision`, `post-tool-record`, `dirty-baseline`, `protected-baseline`, `fdr-import`, `fdr-cycle`, `approval-block`, `approval-consume`, `optout-approval`, `nested-token`, `worker-context-pack`, `worker-invocation`, or `worker-result`. `owner_hash` is SHA-256 over canonical owner JSON with only `owner_hash` set to an empty string. Missing fields, extra fields, tuple mismatch, malformed hash, or timeout parse failure makes the lock owner untrusted. Hooks must not break a global or session lock automatically while a pending install transaction, pending approval transaction, or in-progress trusted import could still be active; enforcing events fail closed with remediation. A repair command may remove a stale lock only outside provider tool execution after verifying the owner process is gone, no pending transaction marker can still be completed, and protected-state ledger continuity is intact.
- Hook self-timeout does not commit partial trusted state. If a deadline fires after a lock is acquired but before all intended JSON/JSONL writes, ledger records, hash-chain updates, and baseline mutations are re-read and verified, the transaction is abandoned as untrusted. Enforcing pre-tool and permission events emit the fixture-verified block/deny timeout decision when that output contract exists; enforcing Stop emits the fixture-verified block/continuation timeout decision. Otherwise enforcing activation for that event fails. Non-enforcing events may return provider-compatible allow/no-op only if they do not publish trusted partial evidence. Any stale lock left by a timed-out hook is handled by the same repair-only stale-lock path; later hooks fail closed while the lock remains unverified.
- Bypass files are one-shot, hash-scoped, audited, and accepted only with a matching exact-schema `bypass-log.jsonl` approval record whose provider, session key, raw session hash, approval hash, pending record hash, `next_user_prompt_marker`, prompt sequence, cwd, project dir, and `source=user-prompt-hook` match the marker. A bypass file by itself is never sufficient.
- A previous-turn bypass may be consumed only when both the bypass file and its matching audit record existed in the turn baseline. Agent-created same-turn bypass files are ignored.
- `turn-baseline-<provider>-<sid>.json` must include approval evidence snapshots needed to prove previous-turn state: confirmation marker fingerprints, bypass marker fingerprints, pending record fingerprints, matching `destructive-log.jsonl`, `bypass-log.jsonl`, and `optout-log.jsonl` byte offsets, last record hashes, and per-record hashes for approval records that can be consumed in the next turn. Pre-existing confirmation or bypass state is consumable only when both marker fingerprint and matching audit record hash were present in the previous trusted `turn-baseline` and the marker/audit tuple matches provider, session key, raw session hash, cwd, project dir, approval hash, pending record hash, `next_user_prompt_marker`, and approval prompt sequence. Records created by the current `user-prompt-submit` hook may be consumed immediately on retry only when the same locked transaction created the marker, appended the audit record with `source=user-prompt-hook`, and updated the approval evidence snapshot.
- Logs must not include large full prompts or full source files by default.
- State directories must be created with `umask 077`, directory mode `0700`, and file mode `0600`. Existing files with broader permissions must be chmod'ed before append. If permissions cannot be corrected for any trusted state file, including baselines, ledgers, checkpoint logs, sequence files, tool-intent logs, permission-decision logs, tool logs, edit logs, FDR artifacts, FDR cycle logs, pending records, confirmation markers, bypass markers, consumed tombstones, nested tokens, installer transaction markers, installer backup manifests, or audit logs used for approvals, that state is untrusted and the relevant gate fails closed. Diagnostic-only logs may degrade to warn/log-only, but they must not be used as trusted evidence while permissions are unsafe.
- `fired-hashes-*`, `hook-errors-*`, `unknown-payloads.jsonl`, and `judge-stderr.log` are diagnostic-only. They may suppress duplicate warnings or aid fixture capture, but they are never approval evidence, freshness evidence, protected-root evidence, or a reason to allow a Stop/pre-tool decision. Missing, truncated, or unsafe diagnostic logs must not weaken any gate.

Trusted JSON canonicalization contract:

- Every trusted JSON or JSONL record, including schema/matrix metadata files, manifests, baselines, ledgers, checkpoint records, sequence files, prompt events, tool-intent/permission-decision/tool/edit logs, approval records, markers, tombstones, FDR artifacts, FDR cycle records, nested tokens, install transaction markers, and backup manifests, must be parsed with duplicate object key rejection before schema validation or hash computation. A parser that silently keeps the first or last duplicate key is not acceptable for trusted state.
- Canonical JSON for hashes is UTF-8, object keys sorted by Unicode code point, no insignificant whitespace, strings escaped deterministically, arrays preserved in the schema-defined order, integers encoded in base-10 without leading zeroes, and no floats unless a specific schema field explicitly allows a bounded decimal. Timestamps are RFC 3339 UTC strings. Paths are normalized before hashing according to the path contract for the field.
- Trusted JSON writers must write their own canonical output for hash-bearing records. Trusted JSON readers must reject duplicate keys, extra fields, missing fields, wrong primitive types, non-finite numbers, and integer values outside the schema range across all trusted JSON. Hash-bearing trusted records additionally reject non-canonical hash strings and hash mismatch. Diagnostic-only JSON may be best-effort, but it cannot feed approval, freshness, protected-root, or allow decisions.

Baseline schema contract:

- `session-baseline-*`, `turn-baseline-*`, `dirty-baseline-*`, and `protected-baseline-*` are exact-schema JSON files with no extra fields, written only under the session state lock with temp-file plus atomic rename and trusted-state ledger coverage. Each baseline has `schema_version` (`1`), `kind`, `provider`, `session_key`, `raw_session_hash`, `cwd`, `project_dir`, `created_at`, `updated_at`, and `baseline_hash`; `kind` is one of `session-baseline`, `turn-baseline`, `dirty-baseline`, or `protected-baseline` and must match the filename class. `baseline_hash` is SHA-256 over canonical baseline JSON with only `baseline_hash` set to an empty string. Missing fields, extra fields, tuple mismatch, kind/filename mismatch, malformed hashes, stale ledger coverage, or hash mismatch makes the baseline untrusted.
- `session-baseline-*` additionally contains `prompt_seq`, `turn_marker`, initial tool-intent/permission-decision/tool/edit log byte offsets, initial tool-intent/permission-decision/tool/edit sequence numbers, initial project opt-out fingerprints, and initial approval evidence snapshot.
- `turn-baseline-*` additionally contains `prompt_seq`, `turn_marker`, previous session baseline hash, current tool-intent/permission-decision/tool/edit log byte offsets, current last tool-intent/permission-decision/tool/edit sequence numbers, last safe allowed Stop sequence boundaries, `last_allow_side_audit_batch_hash`, unresolved blocked Stop scope hashes, approval evidence snapshot, previous-baseline project opt-out fingerprints, and current-capture project opt-out fingerprints. Unresolved blocked Stop scope hashes are the `record_hash` values of exact-schema `bypass-log.jsonl` `blocked` records that remain unresolved for the same provider/session/cwd/project.
- `dirty-baseline-*` additionally contains the git worktree root, git head when available, tracked/staged/untracked/ignored/deleted path records with status and fingerprints, submodule summary records, dirty enumeration limits, and the tool-intent/tool/edit sequence boundary used for the snapshot. A missing or untrusted dirty baseline disables git dirty-snapshot fallback and blocks shell or unknown write-like edit scope.
- `protected-baseline-*` additionally contains protected install baseline hash, protected root aliases, provider config fingerprints, protected runtime/config/pattern/allowlist fingerprints, protected file `dev+inode` index, and mutable trusted-state ledger offsets. Missing or untrusted protected baseline blocks Stop after provider tool execution and blocks trusted approvals/imports.
- Baseline nested fields use exact object names and no extras. Log byte offsets are stored in `log_offsets`, an exact object with `tool_intent`, `permission_decision`, `tool`, and `edit` entries; each entry has `path`, `byte_offset`, `last_seq`, `last_hash`, and `file_fingerprint`. Sequence snapshots are stored in `last_sequences`, an exact object with non-negative integer fields `tool_intent`, `permission_decision`, `tool`, and `edit`. Safe Stop boundaries are stored in `last_safe_stop_sequences` with the same shape. `last_allow_side_audit_batch_hash` is 64 zeroes when the last safe boundary advance required no resolved or consumed-bypass audit records; otherwise it is SHA-256 over canonical JSON with no extras containing `provider`, `session_key`, `raw_session_hash`, `cwd`, `project_dir`, sorted `resolved_record_hashes`, sorted `consumed_bypass_record_hashes`, and sorted `fdr_bypassed_cycle_record_hashes`. Approval evidence is stored in `approval_evidence`, an exact object with `confirmation_markers`, `bypass_markers`, `pending_records`, `destructive_log`, `bypass_log`, and `optout_log`; marker and pending arrays contain exact file fingerprints plus approval/pending/log record hashes, while log entries contain byte offset, last record hash, and per-record hashes for consumable records. Project opt-out fingerprints are stored in `project_optouts_previous` and `project_optouts_current` arrays of exact objects with `kind`, `path`, `realpath`, `fingerprint`, and `source` (`"session-baseline"`, `"turn-baseline"`, or `"approval-log"`).
- Dirty baseline path records are exact objects with `path`, `status` (`"tracked"`, `"staged"`, `"untracked"`, `"ignored"`, `"deleted"`, or `"submodule"`), `old_path`, `fingerprint`, and `old_fingerprint`. `submodule_records` entries contain `path`, `head`, `status`, and `dirty_state_hash`. Dirty enumeration limits are stored in `dirty_limits` with exact non-negative integer fields `max_paths`, `max_ignored_paths`, and `truncated`; `truncated` is `0` or `1`. Protected baseline fingerprint arrays use the trusted ledger fingerprint schema and exact `path`, `realpath`, `kind`, and `fingerprint` fields; protected inode index entries contain `dev`, `inode`, `path`, and `kind`. Any missing nested field, extra nested field, invalid enum, non-canonical path, malformed fingerprint, or mismatch with current trusted logs makes the baseline untrusted.

User prompt sequence contract:

- `seq-<provider>-<sid>.json` is an exact-schema JSON file with no extra fields: `schema_version` (`1`), `provider`, `session_key`, `raw_session_hash`, `cwd`, `project_dir`, `last_seq_by_record_type`, `last_record_hash_by_record_type`, `created_at`, `updated_at`, and `sequence_hash`. `last_seq_by_record_type` is an exact object with non-negative integer fields `tool_intent`, `permission_decision`, `tool`, and `edit`. `last_record_hash_by_record_type` is an exact object with lowercase SHA-256 hash fields `tool_intent`, `permission_decision`, `tool`, and `edit`; a record type with sequence `0` uses 64 zeroes. Each nonzero sequence counter must match the last trusted sequence and hash in the corresponding JSONL log; for `tool_intent` this hash is `intent_hash`, and for `permission_decision`, `tool`, and `edit` it is `record_hash`. `sequence_hash` is SHA-256 over canonical sequence JSON with only `sequence_hash` set to an empty string. Missing fields, extra fields, tuple mismatch, negative or non-integer counters, counter regression, last-record mismatch, malformed hash, ledger mismatch, or hash mismatch makes sequence state untrusted.
- `prompt-seq-<provider>-<sid>.json` is an exact-schema JSON file with no extra fields: `schema_version` (`1`), `provider`, `session_key`, `raw_session_hash`, `cwd`, `project_dir`, `last_prompt_seq`, `last_prompt_event_hash`, `created_at`, `updated_at`, and `sequence_hash`. `last_prompt_seq` is a non-negative integer. `last_prompt_event_hash` is 64 zeroes when `last_prompt_seq` is `0` and otherwise equals the last trusted `prompt-events-*` record hash. `sequence_hash` is SHA-256 over canonical prompt sequence JSON with only `sequence_hash` set to an empty string. Missing fields, extra fields, tuple mismatch, counter regression, prompt-event mismatch, malformed hash, ledger mismatch, or hash mismatch makes prompt sequence state untrusted.
- Every trusted `user-prompt-submit` event allocates a monotonic `prompt_seq` under the session state lock before approval phrase parsing, baseline capture, or pending opt-out detection.
- `prompt-events-<provider>-<sid>.jsonl` records have exactly these fields and no extras: `schema_version` (`1`), `prompt_seq`, `provider`, `session_key`, `raw_session_hash`, `cwd`, `project_dir`, `turn_marker`, `payload_hash`, `ts`, `previous_record_hash`, and `record_hash`. `payload_hash` is SHA-256 over the redacted/truncated normalized prompt payload; the full prompt text is not stored. `previous_record_hash` is 64 zeroes for the first prompt event record and otherwise equals the preceding trusted prompt event record hash for the same provider/session log. `record_hash` is SHA-256 over canonical JSON with only `record_hash` set to an empty string. Hash-chain gaps, duplicate prompt sequences, non-monotonic prompt sequences, tuple mismatches, or prompt-sequence state mismatch make prompt event state untrusted.
- `next_user_prompt_marker` in pending approval records is exactly `prompt-seq:<n>`, where `<n>` is the next prompt sequence after the transaction that created the pending record. A `user-prompt-submit` with `prompt_seq < n` cannot approve it, `prompt_seq == n` must either create the matching approval marker or expire the pending record, and `prompt_seq > n` treats the pending record as expired. If prompt sequence allocation or prompt-event append cannot be verified, approval creation fails closed.

Logging policy:

- Default diagnostic logs store only provider, logical event, session id, tool name/kind, path list, command hash, payload field names, and error reason.
- Raw payload capture is opt-in via protected runtime config `STRICT_CAPTURE_RAW_PAYLOADS=1`.
- Even in raw capture mode, values of keys matching `token|secret|password|authorization|cookie|api[_-]?key` are replaced with `[REDACTED]`.
- String values are truncated to `STRICT_LOG_VALUE_MAX_BYTES` (default 4096).
- Full file contents, full prompts, and full assistant messages are not logged unless `STRICT_CAPTURE_FULL_TEXT=1` is also set.
- `STRICT_CAPTURE_FULL_TEXT=1` does not permit logging provider transcript files, raw provider session/history files, raw transcript slices, or `turn.assistant_text` extracted for FDR challenge. Those values may be read only for fixture-proven extraction and judge prompt construction under the FDR challenge contract; diagnostic logs may record only provider/session tuple, extraction source kind, byte length, cap/truncation reason, and a redacted extraction error code. The current-turn assistant text itself must not be written to trusted state or diagnostic logs.
- `STRICT_CAPTURE_RAW_PAYLOADS`, `STRICT_CAPTURE_FULL_TEXT`, `STRICT_LOG_VALUE_MAX_BYTES`, `STRICT_LOG_MAX_BYTES`, and `STRICT_LOG_MAX_AGE_DAYS` are protected runtime settings. Provider tool environment, project files, shell exports, and prompt text cannot enable broader logging for the current turn. Broader capture is accepted only from generated protected hook config or protected `~/.strict-mode/config/runtime.env` after protected-baseline verification.
- Fixture capture must print the target path before writing and must never append to a world-readable file.

Runtime config policy:

- `~/.strict-mode/config/runtime.env` is optional, protected config. It is parsed by strict-mode code and must never be shell-sourced or eval'ed.
- Format is UTF-8 text, one `KEY=VALUE` per line, blank lines and `#` comments allowed. Keys must be in the runtime whitelist, values are unquoted scalar strings, and no `$VAR`, backtick, command substitution, glob, escape, multiline, or include syntax is interpreted. Duplicate keys make `runtime.env` invalid for that key; first-key/last-key parser behavior must not affect protected settings.
- Whitelisted runtime keys parsed by v0 are `STRICT_NO_ARTIFACT_GATE` (discovery-only; rejected for Phase 5/enforcing v0 as defined in Stop Guard), `STRICT_CAPTURE_RAW_PAYLOADS`, `STRICT_CAPTURE_FULL_TEXT`, `STRICT_LOG_VALUE_MAX_BYTES`, `STRICT_LOG_MAX_BYTES`, `STRICT_LOG_MAX_AGE_DAYS`, `STRICT_CONFIRM_MAX_AGE_SEC`, `STRICT_CONFIRM_MIN_AGE_SEC`, `STRICT_FDR_SOURCE_MAX_BYTES`, `STRICT_DIRTY_IGNORED_MAX_FILES`, `STRICT_CONFIG_LINE_MAX_BYTES`, `STRICT_CLAUDE_JUDGE_MODEL`, `STRICT_CODEX_JUDGE_MODEL`, `STRICT_JUDGE_TIMEOUT_SEC`, `STRICT_CLAUDE_WORKER_MODEL`, `STRICT_CODEX_WORKER_MODEL`, `STRICT_WORKER_TIMEOUT_SEC`, `STRICT_WORKER_CONTEXT_MAX_BYTES`, `STRICT_WORKER_RESULT_MAX_BYTES`, `STRICT_NO_HAIKU_JUDGE`, `STRICT_NO_CODEX_JUDGE`, `STRICT_NO_CLAUDE_WORKER`, `STRICT_NO_CODEX_WORKER`, and `STRICT_LEGACY_CLAUDE_OPTOUTS`.
- Runtime integer bounds are exact: `STRICT_LOG_VALUE_MAX_BYTES` `256..65536` (default `4096`), `STRICT_LOG_MAX_BYTES` `1048576..104857600` (default `10485760`), `STRICT_LOG_MAX_AGE_DAYS` `1..365` (default `30`), `STRICT_CONFIRM_MAX_AGE_SEC` `30..86400` (default `600`), `STRICT_CONFIRM_MIN_AGE_SEC` `0..60` (default `5`), `STRICT_FDR_SOURCE_MAX_BYTES` `65536..10485760` (default `1048576`), `STRICT_DIRTY_IGNORED_MAX_FILES` `0..100000` (default `5000`), `STRICT_CONFIG_LINE_MAX_BYTES` `256..65536` (default `4096`), `STRICT_JUDGE_TIMEOUT_SEC` `5..300` (default `50`), `STRICT_WORKER_TIMEOUT_SEC` `5..300` (default `50`), `STRICT_WORKER_CONTEXT_MAX_BYTES` `4096..1048576` (default `65536`), and `STRICT_WORKER_RESULT_MAX_BYTES` `1024..262144` (default `16384`). `STRICT_CONFIRM_MIN_AGE_SEC` must not exceed `STRICT_CONFIRM_MAX_AGE_SEC`.
- `STRICT_NO_CLAUDE_WORKER` and `STRICT_NO_CODEX_WORKER` default to `1` in generated runtime config because real worker invocation requires installed-version `worker-invocation` fixture proof. In the discovery skeleton, `STRICT_NO_CLAUDE_WORKER=0` or `STRICT_NO_CODEX_WORKER=0` is invalid because there is no proof-bound worker enable path yet. A provider worker may be enabled only after that proof exists and is protected by the install manifest and protected install baseline.
- Boolean values must be exactly `0` or `1`; integer values must be base-10 and within documented bounds; enum/model values must match the command construction whitelist for the relevant consumer. Unknown keys or invalid values make runtime config invalid and the affected protected setting falls back to the safe default while logging a config error. Settings that weaken enforcement, logging privacy, or approval TTLs fail closed instead of silently accepting an invalid value.
- Runtime settings resolution order is: safe default, protected `runtime.env`, generated protected hook config. A provider tool's process environment is diagnostic only for these keys unless it exactly matches a generated protected hook config value whose containing provider config fingerprint matches the protected baseline.

State cleanup:

- Expired `pending-destructive-*`, `confirm-*`, `pending-bypass-*`, `bypass-*`, and `pending-optout-*` files are ignored even if audited deletion fails. Deletion of expired pending records or markers is trusted only when the cleanup or first-next-prompt expiry transaction appends the matching `expired` audit record plus trusted-state ledger mutation; an unaudited disappearance is protected-state forgery. Consumed tombstones are not approval evidence, but v0 has no checkpoint schema for approval audit logs or tombstone marker files; cleanup and rotation must retain consumed tombstones and approval audit records rather than compacting them.
- Cleanup runs on `session-start`, `user-prompt-submit`, and `stop`.
- Pending approval records store creation timestamp and the exact `next_user_prompt_marker` from the prompt sequence contract. The first `user-prompt-submit` after a pending record either consumes the exact approval phrase or expires the pending record.
- `STRICT_CONFIRM_MAX_AGE_SEC` bounds pending records, confirmations, bypasses, and the pending opt-out approval window before an approved audit record is created. Default is 600 seconds. Approved opt-out audit records do not expire by wall-clock TTL; they activate only the exact current file fingerprint/path/permissions tuple and become inactive when that tuple changes, disappears, or becomes unsafe.
- Logs use size/age rotation (`STRICT_LOG_MAX_BYTES`, `STRICT_LOG_MAX_AGE_DAYS`) and rotation must not remove active `seq-*`, `prompt-seq-*`, `prompt-events-*`, `tools-*`, `tool-intents-*`, `permission-decisions-*`, `edits-*`, `session-baseline-*`, `turn-baseline-*`, `dirty-baseline-*`, `protected-baseline-*`, `trusted-state-ledger-*`, `trusted-state-ledger-global.jsonl`, `fdr-cycles-*`, nested judge token records, pending records, confirmation/bypass/opt-out evidence, approval audit records, unresolved blocked Stop scope evidence, or trusted FDR artifacts for the current session. Rotation may compact old checkpointable logs/ledgers only after a Stop decision has completed and no pending approval/import/challenge or unresolved blocked Stop scope for that session references the evidence. Compaction of trusted ledgers, prompt event logs, permission-decision logs, or FDR cycle logs requires exact-schema `state.checkpoint.v1` records in `checkpoints-<provider>-<sid>.jsonl` or `checkpoints-global.jsonl`; plain truncation or deletion makes the affected trusted state untrusted. Approval audit logs, consumed tombstones, baseline files, and trusted FDR artifact files are exact evidence, not checkpointable log streams in v0; rotation must not checkpoint, truncate, or delete them. Active `seq-*` and `prompt-seq-*` state files are not compacted; they advance monotonically for the session and deletion makes current-turn boundaries untrusted.
- Checkpoint records have exactly these fields and no extras: `schema_version` (`1`), `checkpoint_scope` (`"session"` or `"global"`), `provider`, `session_key`, `raw_session_hash`, `cwd`, `project_dir`, `checkpointed_path`, `checkpointed_kind`, `covered_seq_min`, `covered_seq_max`, `covered_byte_min`, `covered_byte_max`, `last_covered_record_hash`, `retained_successor_record_hash`, `source_ledger_record_hash`, `created_at`, `checkpoint_hash`, `previous_record_hash`, and `record_hash`. Session checkpoints must populate the full provider/session/raw-session/cwd/project tuple; global checkpoints must use empty tuple fields. Approval audit logs are not checkpointable in v0, so no global checkpoint may carry a session tuple by referencing approval evidence. `checkpointed_kind` is one of `prompt-event-log`, `permission-decision-log`, `ledger`, or `fdr-cycle-log`. Covered ranges are inclusive non-negative integers; empty coverage is invalid. `last_covered_record_hash`, `retained_successor_record_hash`, `source_ledger_record_hash`, `checkpoint_hash`, `previous_record_hash`, and `record_hash` are lowercase SHA-256 hashes, with `retained_successor_record_hash` set to 64 zeroes only when no successor record remains. `checkpoint_hash` is SHA-256 over canonical JSON containing the checkpoint scope tuple, checkpointed path/kind, covered ranges, last covered record hash, retained successor record hash, and source ledger record hash. `record_hash` is SHA-256 over canonical checkpoint record JSON with only `record_hash` set to an empty string. A checkpoint is trusted only when the source ledger record is already present in the trusted ledger hash chain, has `writer="repair"`, `operation="checkpoint"`, `related_record_hash` set to 64 zeroes, a checkpointable `target_class`, old/new fingerprints that match the checkpointed path before and after compaction, matching target path/kind/scope tuple, and no pending approval/import/challenge or unresolved blocked Stop scope references the covered range.
- Cleanup must never delete files outside the resolved state root.

Approval hash schema:

- Every user-visible `<hash>` in `strict-mode confirm <hash>`, `strict-mode bypass <hash>`, and `strict-mode approve-optout <hash>` is a full 64-character lowercase hex SHA-256 digest.
- Approval phrase parsing is ASCII-only. The hook trims only leading/trailing spaces and tabs from each physical prompt line, then requires exactly `strict-mode confirm <hash>`, `strict-mode bypass <hash>`, or `strict-mode approve-optout <hash>` with one ASCII space between tokens. Quotes, markdown fences, punctuation, localized aliases, extra tokens, zero-width characters, non-ASCII whitespace, or surrounding prose do not approve anything.
- The digest input is canonical JSON encoded as UTF-8 with sorted keys, no insignificant whitespace, normalized absolute paths, provider, session key, raw session hash, cwd, project dir where relevant, block class, and the specific immutable payload being approved. Timestamps, TTL fields, log paths, and mutable file mtimes are excluded from the hash.
- Destructive confirmation hashes bind to the normalized shell command hash, provider, session key, raw session hash, cwd, and project dir. Quality bypass hashes bind to block class, block reason hash, edited path list hash, current-turn tool-intent/tool/edit sequence lists, tool-intent/tool/edit log digests, content-scope fingerprint digest, `gate_context`, provider, session key, raw session hash, cwd, and project dir. `gate_context` is an object with exact schema per gate and must contain gate-specific immutable subject fields. For FDR challenge bypasses it contains `gate` (`"fdr-challenge"`), `scope_digest`, `artifact_hash`, and `fdr_cycle_record_hash`. For FDR artifact validation bypasses it contains `gate` (`"fdr-artifact"`), `artifact_state` (`"missing"`, `"invalid"`, `"stale"`, or `"incomplete"`), `artifact_hash`, `artifact_verdict`, and `artifact_error_hash`; missing artifacts use 64 zeroes for `artifact_hash`. For stub/static/trivial quality blocks it contains `gate`, `finding_digest`, and `finding_count`, where `finding_digest` is SHA-256 over the exact normalized finding/warning list. A non-FDR quality bypass without the required gate-specific subject fields is untrusted. Opt-out hashes bind to opt-out path, file fingerprint, provider, session key, raw session hash, cwd, and project dir.
- `command_hash`, `block_reason_hash`, `edited_path_list_hash`, and file content fingerprints are also lowercase SHA-256 hex digests over canonical normalized inputs. For `command_hash_source="shell-string"`, `command_hash` is SHA-256 over canonical JSON containing the normalized shell command string, provider, session key, raw session hash, cwd, project dir, and `command_hash_source`. For `command_hash_source="argv"` or `"trusted-import-argv"`, it is SHA-256 over canonical JSON containing the exact argv array plus provider, session key, raw session hash, cwd, project dir, and `command_hash_source`. `block_reason_hash` is SHA-256 over canonical JSON containing `gate`, `block_class`, stable `reason_code`, redacted rendered reason text, provider, session key, raw session hash, cwd, and project dir. `edited_path_list_hash` is SHA-256 over canonical JSON containing sorted normalized project-relative `edited_paths`, `deleted_paths`, and `renamed_paths` after dirty-snapshot merge. `content_scope_fingerprint_digest` is SHA-256 over canonical JSON containing sorted entries for each existing edited path with `path` plus the full trusted ledger fingerprint object, including `kind`, `link_target`, and `tree_hash`, plus deletion and rename markers for non-existing or moved paths. `command_hash` values may be compared only when `command_hash_source` matches. Destructive approvals use `command_hash_source="shell-string"`; trusted FDR import provenance uses `command_hash_source="trusted-import-argv"`. If canonicalization fails or two pending records collide on the same visible hash, the hook must block and ask for a regenerated pending record rather than accepting approval.

Pending approval record schema:

- `pending-destructive-*`, `pending-bypass-*`, and `pending-optout-*` files are JSON with exact schemas and no extra fields. The filename `<hash>` is `approval_hash`, not `pending_record_hash`.
- Common fields for every pending approval record are `schema_version` (`1`), `kind` (`"destructive-confirm"`, `"quality-bypass"`, or `"optout-approval"`), `provider`, `session_key`, `raw_session_hash`, `cwd`, `project_dir`, `approval_hash`, `created_at`, `expires_at`, `next_user_prompt_marker`, and `pending_record_hash`.
- `pending_record_hash` is SHA-256 over canonical pending-record JSON with only `pending_record_hash` set to an empty string. `approval_hash` is the user-visible hash computed by the Approval hash schema and excludes timestamps, TTL fields, log paths, and mutable mtimes. The pending filename hash, `approval_hash`, provider, session key, raw session hash, cwd, project dir, and `pending_record_hash` must all match before any user prompt can approve the record.
- `next_user_prompt_marker` must match the exact `prompt-seq:<n>` value allocated by the prompt sequence contract. A pending record with an empty, malformed, stale, or future-skipping marker is untrusted and cannot create an approval marker.
- `pending-destructive-*` adds exactly `normalized_command`, `command_hash`, `command_hash_source`, and `block_reason_hash`; `command_hash_source` is exactly `shell-string`.
- `pending-bypass-*` adds exactly `block_class`, `block_reason_hash`, `edited_path_list_hash`, `tool_intent_seq_list`, `tool_seq_list`, `edit_seq_list`, `tool_intent_log_digest`, `tool_log_digest`, `edit_log_digest`, `content_scope_fingerprint_digest`, and `gate_context`.
- `pending-optout-*` adds exactly `optout_path`, `optout_realpath`, `optout_kind` (`"disabled"`, `"no-static-prepass"`, `"no-destructive-gate"`, or `"legacy-claude"`), and `optout_fingerprint`. `optout_fingerprint` is SHA-256 over canonical JSON containing `optout_realpath`, `owner_uid`, and the full trusted ledger fingerprint object after protected path checks. The fingerprint kind must be `file`; symlink, directory, missing, group/other-writable, owner-mismatched when ownership is exposed, or unreadable-permission opt-out files are untrusted and cannot create pending or approved opt-out records.
- A pending record whose filename hash, exact fields, canonical hash, expiry, next-user-prompt marker, provider/session/raw-session/cwd/project tuple, protected permissions, or ledger mutation cannot be verified is ignored and cannot create a marker.

Approval audit record schema:

- `destructive-log.jsonl`, `bypass-log.jsonl`, and `optout-log.jsonl` are append-only JSONL audit logs. Each record has these common fields: `schema_version` (`1`), `log` (`"destructive"`, `"bypass"`, or `"optout"`), `action`, `provider`, `session_key`, `raw_session_hash`, `cwd`, `project_dir`, `approval_hash`, `pending_record_hash`, `next_user_prompt_marker`, `prompt_seq`, `source`, `ts`, `previous_record_hash`, and `record_hash`. `prompt_seq` is the allocated prompt sequence for `source=user-prompt-hook`; otherwise it is `0`. No fields are allowed beyond the common fields plus the log-specific fields in the audit action and source bullets in this file.
- `record_hash` is SHA-256 over canonical audit record JSON with only `record_hash` set to an empty string. `previous_record_hash` is the previous record hash for the same audit log or 64 zeroes for the first record. Hash-chain gaps, duplicate hashes, malformed records, or provider/session/raw-session/cwd/project mismatches make the affected approval evidence untrusted.
- Destructive audit actions are `blocked`, `confirmed`, `consumed`, and `expired`; destructive records also include `command_hash` and `command_hash_source`. Bypass audit actions are `blocked`, `approved`, `consumed`, `resolved`, and `expired`; bypass records also include `block_class`, `block_reason_hash`, `edited_path_list_hash`, `tool_intent_seq_list`, `tool_seq_list`, `edit_seq_list`, `tool_intent_log_digest`, `tool_log_digest`, `edit_log_digest`, `content_scope_fingerprint_digest`, and `gate_context`. Opt-out audit actions are `pending`, `approved`, and `expired`; opt-out records also include `optout_path`, `optout_realpath`, `optout_kind`, and `optout_fingerprint`.
- `destructive-log.jsonl` and `bypass-log.jsonl` `consumed` records also include exactly `marker_hash`, `active_marker_path`, `consumed_tombstone_path`, `marker_pre_rename_fingerprint`, and `tombstone_fingerprint`. The two fingerprint objects use the full trusted ledger fingerprint schema. `active_marker_path` must be the canonical active marker path that was renamed, `consumed_tombstone_path` must be the canonical tombstone path, `marker_pre_rename_fingerprint` must match the active marker immediately before rename, and `tombstone_fingerprint` must match the tombstone immediately after rename. The tombstone file must parse as the exact preserved marker JSON and its `marker_hash` must equal the consumed audit record's `marker_hash`. Non-`consumed` audit actions must not include these fields.
- `bypass-log.jsonl` `resolved` records also include exactly `resolved_tool_intent_seq_list`, `resolved_tool_seq_list`, `resolved_edit_seq_list`, `resolved_tool_intent_log_digest`, `resolved_tool_log_digest`, `resolved_edit_log_digest`, and `resolved_content_scope_fingerprint_digest`. These fields describe the current superseding scope that passed all quality gates without using a bypass. Non-`resolved` audit actions must not include these fields. The turn-baseline mutation that advances `last_safe_stop_sequences` for one or more resolved/consumed quality scopes must set `last_allow_side_audit_batch_hash` to the canonical allow-side audit batch hash for all resolved, consumed-bypass, and gate-specific `bypassed` cycle records required by that allow. Its trusted-state ledger record must use `target_class="baseline"`, `operation="modify"`, `target_path` equal to the canonical `turn-baseline-*` path, and `related_record_hash` equal to that same batch hash.
- `source` is `pre-tool-hook`, `stop-hook`, `user-prompt-hook`, or `cleanup`, and action/source pairs are closed. Destructive `blocked` records must use `source=pre-tool-hook`; destructive `confirmed` must use `source=user-prompt-hook`; destructive `consumed` must use `source=pre-tool-hook`; destructive `expired` may use only `source=user-prompt-hook` for first-next-prompt expiry or `source=cleanup` for TTL cleanup. Bypass `blocked` and `resolved` records must use `source=stop-hook`; bypass `approved` must use `source=user-prompt-hook`; bypass `consumed` must use `source=stop-hook`; bypass `expired` may use only `source=user-prompt-hook` for first-next-prompt expiry or `source=cleanup` for TTL cleanup. Opt-out `pending` records may use only `source=user-prompt-hook` or `source=stop-hook`; opt-out `approved` must use `source=user-prompt-hook`; opt-out `expired` may use only `source=user-prompt-hook` for first-next-prompt expiry or `source=cleanup` for TTL cleanup. A `confirmed` or `approved` record used to create a confirmation marker, bypass marker, or opt-out approval must reference the exact pending record hash, `next_user_prompt_marker`, prompt sequence, and provider/session/raw-session/cwd/project tuple captured in trusted state. Invalid action/source pairs make the audit record untrusted.

Approval marker and tombstone schema:

- `confirm-<provider>-<sid>-<hash>` and `bypass-<provider>-<sid>-<hash>` marker files are JSON, not empty sentinel files. Each marker has exactly these fields and no extras: `schema_version` (`1`), `kind` (`"destructive-confirm"` or `"quality-bypass"`), `provider`, `session_key`, `raw_session_hash`, `cwd`, `project_dir`, `approval_hash`, `pending_record_hash`, `next_user_prompt_marker`, `approval_prompt_seq`, `approval_log_record_hash`, `source` (`"user-prompt-hook"`), `created_at`, `expires_at`, and `marker_hash`.
- `marker_hash` is SHA-256 over canonical marker JSON with only `marker_hash` set to an empty string. A marker whose filename hash, `approval_hash`, pending record hash, audit record hash, provider, session key, raw session hash, cwd, project dir, `next_user_prompt_marker`, `approval_prompt_seq`, source, expiry, permissions, or marker hash does not match is untrusted and cannot authorize allow.
- Consumption never deletes an active marker directly. Under the required state locks, `strict-hook` atomically renames `confirm-*` to `consumed-confirm-*.json` or `bypass-*` to `consumed-bypass-*.json`, preserving the exact marker bytes, then appends the consumed audit record and trusted-state ledger mutation before emitting allow. The ledger record for the tombstone mutation must use `target_class="consumed-tombstone"`, `operation="rename"`, `target_path` equal to `consumed_tombstone_path`, `old_fingerprint` for a missing tombstone, `new_fingerprint` equal to the consumed audit record's `tombstone_fingerprint`, and `related_record_hash` equal to the consumed audit `record_hash`; verification also requires the consumed audit `active_marker_path` to be absent after the rename. If the active marker still exists after the rename, the tombstone path already exists, the consumed audit append cannot be verified, or the matching ledger record is missing or mismatched, the gate fails closed.
- A consumed tombstone is not approval evidence. Any later active marker with the same approval hash is ignored unless a new pending record and a new user-prompt-hook audit record produce a different approval hash.
