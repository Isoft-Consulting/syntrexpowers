# 3. Hook Event Matrix

Part of [Strict Mode Universal Experimental - Specification v0](../SPEC.md).


### 3.1 Logical Events

The universal package uses these logical event names:

| Logical event | Purpose |
|---|---|
| `session-start` | health check and state setup |
| `user-prompt-submit` | inject compact strict reminder |
| `pre-tool-use` | block dangerous or stub-producing tool calls before execution |
| `post-tool-use` | record edits and start optional static prepass |
| `stop` | enforce stop-time checks before the agent waits for next user input |
| `subagent-stop` | optional subagent stop handling when provider exposes it |
| `permission-request` | optional permission decision hook when provider exposes it |

### 3.2 Provider Support

| Logical event | Claude Code | Codex CLI | v0 target |
|---|---:|---:|---|
| `session-start` | `SessionStart` | `SessionStart` | target both after fixture smoke |
| `user-prompt-submit` | `UserPromptSubmit` | `UserPromptSubmit` | target both after injection contract proof |
| `pre-tool-use` | `PreToolUse` | `PreToolUse` | target both after payload + block fixtures |
| `post-tool-use` | `PostToolUse` | `PostToolUse` | target both after payload fixtures |
| `stop` | `Stop` | `Stop` | target both after payload + block fixtures |
| `subagent-stop` | `SubagentStop` | no confirmed equivalent | Claude only when fixture-proven in v0 |
| `permission-request` | not required by current bundle | `PermissionRequest` exists in Codex | conditional v0 gate if fixtures show it can approve risky tools |

Codex internal notifications such as `turn/started` are not hook events for this bundle unless proven configurable through `hooks.json`.

“Target both” means the installer may register the hook in discovery/log-only mode first. Fail-closed behavior and prompt-injection output are enabled per event only after that event's payload and decision/output contract fixtures pass.

Discovery/log-only `pre-tool-use` hooks may run provider proof, normalization, protected baseline loading, and shared classifiers before enforcement is enabled. Any such diagnostic result must be recorded as the nested hash-bound `hook.preflight.v1` object, with raw command text, raw path lists, raw classifier reasons, raw baseline errors, and raw payload content omitted from JSONL. A discovery preflight record is not trusted state, cannot approve or deny a tool by itself, and cannot serve as approval, freshness, baseline, or protected-root evidence for later enforcing decisions.

Enforcing activation also requires fixture proof that at least one baseline-capable early event, `session-start` or `user-prompt-submit`, fires before the provider can execute model tool calls for that session. That event must expose enough provider/session/cwd/project identity for strict-mode to write `session-baseline-*`, `turn-baseline-*` when applicable, and `protected-baseline-*` before write-like, shell, permission, import, or Stop evidence is trusted. If no such early event is fixture-proven for the installed provider version/build, the provider remains discovery/log-only or activation fails for enforcing v0.

Provider fixture proof contract:

- Every fixture-proven payload, matcher, command-execution, event-order, prompt-extraction, judge-invocation, and decision-output contract must be recorded in a checked-in exact-schema `fixture-manifest.json` with no extra fields. The top-level manifest has exactly `schema_version` (`1`), `generated_at`, `records`, and `manifest_hash`; `manifest_hash` is SHA-256 over canonical manifest JSON with only `manifest_hash` set to an empty string.
- Each fixture manifest record has exactly these fields and no extras: `schema_version` (`1`), `contract_id`, `provider`, `provider_version`, `provider_build_hash`, `platform`, `event`, `contract_kind`, `payload_schema_hash`, `decision_contract_hash`, `command_execution_contract_hash`, `fixture_file_hashes`, `captured_at`, `compatibility_range`, and `fixture_record_hash`. `contract_id` is a stable lowercase identifier unique within the manifest. `provider` is `claude` or `codex`; `provider_version` is the exact discovered version or `unknown`; `provider_build_hash` is a lowercase SHA-256 hash or an empty string; `contract_kind` is one of `payload-schema`, `matcher`, `command-execution`, `event-order`, `prompt-extraction`, `judge-invocation`, `decision-output`, or `version-comparator`. `event-order` records prove lifecycle timing such as baseline-capable events firing before model tool execution. `payload_schema_hash`, `decision_contract_hash`, and `command_execution_contract_hash` are lowercase SHA-256 hashes when the contract kind proves that surface, and exactly 64 zeroes when that field is not applicable to the contract kind; empty strings are not valid hash sentinels. `fixture_file_hashes` is a sorted array of exact objects with no extra fields: `path` and `content_sha256`; `path` is the normalized repository-relative fixture file path and `content_sha256` is the lowercase SHA-256 hash of the raw fixture file bytes. `compatibility_range` is an exact object with no extra fields: `mode` (`"exact"`, `"range"`, or `"unknown-only"`), `min_version`, `max_version`, `version_comparator`, and `provider_build_hashes`. For `mode="exact"`, `min_version` and `max_version` both equal `provider_version` and `version_comparator` is empty. For `mode="unknown-only"`, `provider_version` must be `unknown`, both version bounds are `unknown`, and `version_comparator` is empty. For `mode="range"`, both version bounds are non-empty provider-native version strings, `version_comparator` names a fixture-proven comparator implemented by strict-mode, and `provider_build_hashes` is a sorted array of lowercase SHA-256 hashes or an empty array when build hashes are unavailable. `fixture_record_hash` is SHA-256 over canonical record JSON with only `fixture_record_hash` set to an empty string.
- Typed `command-execution`, `matcher`, and `event-order` proofs are not trusted as self-attested JSON. `command-execution` records must bind the typed proof to a matching discovery-record fixture plus stdout, stderr, and exit-code fixtures, and the proof payload hash must match a checked-in raw payload fixture for the same provider/event. `matcher` records must bind the typed proof to a matching discovery-record fixture and raw payload fixture. `event-order` records must bind every observed event/payload/timestamp tuple to a matching discovery-record fixture and raw payload fixture. Missing external bindings, stdout/stderr/exit-code drift, discovery-record drift, or payload-hash drift makes the fixture manifest invalid and prevents readiness selection.
- Enforcing activation may use a fixture proof only when the installed provider identity matches `provider`, `provider_version` or the exact `compatibility_range`, platform constraints, event, contract kind, `contract_id`, fixture record hash, and manifest hash. A `range` compatibility proof is invalid unless the named `version_comparator` exists in protected runtime code and its own fixture record is present in the same manifest. If provider version/build cannot be discovered, only fixtures with `compatibility_range.mode="unknown-only"` may enable enforcement; otherwise the affected event stays discovery/log-only or activation fails when that event is required for safety.
- The installer records fixture manifest hashes, selected contract ids, provider version/build evidence, and output contract ids in the install manifest and protected install baseline. At hook startup, provider version/build drift from the protected baseline makes affected fixture-dependent gates untrusted until reinstall/rediscovery; destructive/protected/permission/Stop enforcement must fail closed rather than silently using stale contracts.

Codex `PermissionRequest` handling is not optional if fixture discovery shows it can approve shell, write, patch, network, or filesystem operations independently of `PreToolUse`. In that case universal v0 must register `permission-request`, run the same protected-path/destructive classification before allowing the permission, and append a trusted permission decision record before emitting allow. Deny decisions also attempt to append the same exact-schema record before emission; if deny recording fails, the hook must still emit the fixture-verified provider deny/block response with `deny-record-failure` rather than returning allow. If fixtures prove `PermissionRequest` is informational only or not involved in tool approval, v0 may leave it log-only.

Permission classification:

- Shell/write/edit/multi-edit/patch permission requests run the same destructive and protected-path gates as `pre-tool-use`.
- Filesystem permission requests must expose access mode, recursive mode, and paths. Read-only requests may allow when all resolved paths are outside protected roots, do not match protected file `dev+inode` entries, and are within the current project or an explicitly configured read allowlist. Write/execute/delete/chmod requests are blocked unless every target can be proven non-protected and within the current project. Unknown access mode, unknown recursive mode, unknown target set, home-wide access, root-wide access, or provider-level broad filesystem approval fails closed.
- Network permission requests are deny-by-default in v0 unless a protected config allowlist names the exact scheme, host, port, and operation. Unknown scheme, host, port, or operation, wildcard hosts, credential-bearing URLs, proxy changes, tunneling, listener creation, and broad network approval fail closed.
- If a PermissionRequest combines several capabilities, normalized `permission.operation` is `combined`, all exposed capability-specific fields must still be populated, and the strictest capability decision wins; one blocked capability blocks the whole request.
- Approval-capable `permission-request` events use `permission-decisions-<provider>-<sid>.jsonl`, not `tool-intents-*`, because a native permission decision may not produce a later `post-tool-use` record. Before emitting an enforcing allow, strict-mode must append and verify a permission decision record under the session state lock. Missing, malformed, duplicate, non-monotonic, record-hash-mismatched, or previous-record-hash-mismatched permission decision records make the permission decision untrusted; for approval-capable requests, untrusted allow recording fails closed rather than returning provider allow. A deny decision with untrusted recording remains a provider deny/block and must not become an allow.

Protected config file format:

- `~/.strict-mode/config/destructive-patterns.txt`, `~/.strict-mode/config/protected-paths.txt`, `~/.strict-mode/config/stub-allowlist.txt`, `~/.strict-mode/config/filesystem-read-allowlist.txt`, and `~/.strict-mode/config/network-allowlist.txt` are protected config files and must be included in protected install and session baselines.
- These files are parsed by strict-mode code, never shell-sourced, eval'ed, glob-expanded, or interpreted by provider shells. Encoding is UTF-8. Blank lines and lines starting with `#` are comments. Inline comments are not supported. NUL bytes, non-UTF-8 input, and lines over `STRICT_CONFIG_LINE_MAX_BYTES` (default 4096) make the relevant config file untrusted. For destructive/protected/stub config files, unknown directives, wrong field counts, invalid paths, invalid hashes, or uncompiled patterns make the relevant config file untrusted and any enforcing decision depending on it fails closed. For network/filesystem allowlist files, syntactically invalid allow entries are ignored with a config error and cannot create an allow; duplicate valid allow entries are harmless after canonical de-duplication.
- Protected path lines have exactly one of two forms: `protect-file <absolute-file-path>` or `protect-tree <absolute-directory-path>/**`. For `protect-tree`, `/**` is a grammar suffix, not a glob inside the normalized path. The path must be normalized, absolute, and must not contain shell syntax, glob characters, `..`, symlink components, NUL, newline, or trailing slash ambiguity. Built-in protected roots are always active even if `protected-paths.txt` is empty or untrusted.
- Destructive pattern lines have exactly one of two forms: `shell-ere <POSIX-ERE>` or `argv-token <literal-token>`. `shell-ere` is compiled once by strict-mode code and matched against the normalized shell command string only; it is never passed to grep/sed/awk through a shell. `argv-token` is compared against shell-lexer tokens after quote removal and safe home expansion, without command substitution, glob expansion, variable expansion beyond `~`, `$HOME`, and `${HOME}`, or alias expansion. Invalid destructive pattern config blocks shell enforcement rather than silently dropping the pattern.
- Stub allowlist lines have exactly the form `finding <finding_digest>`, where `finding_digest` is SHA-256 over the scanner's canonical finding object containing normalized project-relative path, marker kind, matched marker text hash, line content hash, and surrounding function/class context hash when available. Line numbers alone are never allowlist identity. An allowlist entry suppresses only that exact finding digest and only when the allowlist file fingerprint was present in the previous trusted turn baseline or was approved through the quality bypass flow.
- Filesystem read allowlist lines have exactly one of two forms: `read <absolute-file-path>` or `read-tree <absolute-directory-path>/**`. For `read-tree`, `/**` is a grammar suffix, not a glob inside the normalized path; validation removes the suffix first, then validates the directory prefix. The file path or directory prefix must be normalized, absolute, outside protected roots, and must not contain shell syntax, glob characters, `..`, symlink components, NUL, newline, or trailing slash ambiguity. The allowlist target and every requested read target covered by it must be checked with `lstat` path-component checks and final `stat`; if any final `dev+inode` matches a protected file, the read request fails closed even when the lexical path is outside protected roots. Directory read allowlisting applies only under that directory after these identity checks. No write/execute/delete/chmod mode can be allowlisted in v0.
- Network allowlist lines are exactly `<operation> <scheme> <host> <port>`, where operation is `connect`, scheme is `http` or `https`, host is a canonical lower-case IDNA A-label hostname or literal IP address, and port is an integer `1..65535`. Wildcards, CIDR ranges, inherited subdomains, userinfo, paths, queries, fragments, proxies, tunneling, listener creation, and environment-derived hosts are not allowed.
- Invalid allowlist lines are ignored with a config error. If a PermissionRequest would rely on an invalid line, it fails closed.
